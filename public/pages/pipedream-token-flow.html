Overview 概述

Pipedream Connect lets your app authorize users to third-party services (GitHub, Gmail, Slack, etc.) without handling OAuth flows yourself. The key mechanism is a connect_token — a short-lived credential that bridges your authenticated user to Pipedream's OAuth popup.

Pipedream Connect 让你的应用能授权用户访问第三方服务（GitHub、Gmail、Slack 等），无需自己处理 OAuth 流程。核心机制是 connect_token — 一个短时效凭证，将你已认证的用户桥接到 Pipedream 的 OAuth 弹窗。

Step-by-Step Details 逐步详解

1
User Clicks "Add Connector" 用户点击添加连接器
User opens the Customize View, goes to the Connectors tab, and clicks on a connector (e.g., GitHub). This triggers handleDirectConnect() which initializes the Pipedream frontend client.
用户打开 Customize View，切到 Connectors 标签页，点击某个连接器（如 GitHub）。触发 handleDirectConnect()，初始化 Pipedream 前端客户端。
components/CustomizeView/ConnectorsTab.tsx
const handleDirectConnect = async (connector) => {
  const client = getPipedreamClient(String(user.id));

  await client.connectAccount({
    app: connector.id,  // e.g. 'github', 'gmail'
    onSuccess: (res) => { /* Step 9 */ },
    onError: (err) => { console.error(err); },
  });
};
2
SDK Triggers tokenCallback SDK 触发令牌回调
When connectAccount() is called, the Pipedream SDK internally needs a valid connect_token to authenticate the iframe. It calls the tokenCallback function you provided during client initialization. This is an automatic internal step — your code doesn't call it directly.
调用 connectAccount() 时，Pipedream SDK 内部需要一个有效的 connect_token 来认证 iframe。它会调用你在初始化客户端时提供的 tokenCallback 函数。这是 SDK 自动执行的内部步骤。
libs/pipedream/client.ts
import { createFrontendClient } from '@pipedream/sdk/browser';

export function getPipedreamClient(userId: string) {
  return createFrontendClient({
    externalUserId: userId,
    // SDK calls this whenever it needs a fresh token
    tokenCallback: async () => {
      const res = await fetch('/api/connectors/token', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
      });
      return res.json();
    },
  });
}
3
POST /api/connectors/token 请求到达服务端
The browser sends a POST request to the Next.js API route. The auth cookie is automatically included by the browser (same-origin). This is where server-side processing begins.
浏览器发送 POST 请求到 Next.js API route。Auth cookie 由浏览器自动携带（同源请求）。服务端处理从这里开始。
app/api/connectors/token/route.ts
export async function POST(request: Request) {
  // Environment variables — never exposed to browser
  const clientId = process.env.PIPEDREAM_CLIENT_ID;
  const clientSecret = process.env.PIPEDREAM_CLIENT_SECRET;
  const projectId = process.env.PIPEDREAM_PROJECT_ID;
  // ... continues in Step 4
}
4
Read Session Cookie 读取会话 Cookie
The server extracts the auth token from the request. It first checks the Authorization header, then falls back to the session cookie. The cookie key varies by environment (Authorization for prod, Authorization-staging for staging). This token proves the user is authenticated.
服务端从请求中提取 auth token。先检查 Authorization header，再回退到 session cookie。Cookie key 因环境而异（生产用 Authorization，staging 用 Authorization-staging）。这个 token 证明用户已认证。
libs/auth/server.ts → requireAuth()
// Step 1: Try Authorization header
const authHeader = request.headers.get('authorization');
if (authHeader?.startsWith('Bearer ')) {
  return authHeader.substring(7);
}

// Step 2: Fall back to cookie
const cookieStore = await cookies();
const token = cookieStore.get(AUTH_COOKIE_KEY)?.value;
// AUTH_COOKIE_KEY = 'Authorization' (prod) | 'Authorization-staging'
5
Resolve Numeric User ID 解析数字用户 ID
Critical step! JWT tokens from OAuth providers (e.g. Google) may have sub values like "google-oauth|12345" which aren't reliable numeric IDs. The server must resolve the real numeric user ID from the backend, because Pipedream uses externalUserId to scope accounts per user. A wrong ID = cross-user account leaks!
关键步骤！OAuth 提供商（如 Google）的 JWT token 的 sub 值可能是 "google-oauth|12345" 这种格式，不是可靠的数字 ID。服务端必须从后端解析出真实的数字用户 ID，因为 Pipedream 用 externalUserId 按用户隔离账户。ID 错误 = 跨用户账户泄漏！
app/api/connectors/token/route.ts (lines 23-43)
// Try parsing from JWT sub field
let userId = parseJwtSub(token); // might return "google-oauth|..."

// If not a clean numeric ID, fetch from backend
if (!userId || isNaN(Number(userId))) {
  const res = await fetch(`${BACKEND_URL}/api/users/me`, {
    headers: { Authorization: `Bearer ${token}` }
  });
  const user = await res.json();
  userId = String(user.id); // Now we have: "42"
}
6
pd.tokens.create() 创建 Connect Token
With the verified user ID, the server initializes the Pipedream backend client (using secret credentials) and calls pd.tokens.create(). This generates a short-lived connect_token scoped to this specific user. The allowedOrigins parameter restricts which domains can use this token (CORS security).
用验证过的用户 ID，服务端初始化 Pipedream 后端客户端（使用密钥凭证）并调用 pd.tokens.create()。生成一个短时效的 connect_token，绑定到特定用户。allowedOrigins 参数限制哪些域名可以使用此 token（CORS 安全）。
app/api/connectors/token/route.ts (lines 51-65)
const pd = new PipedreamClient({
  clientId,      // from env
  clientSecret,  // from env — NEVER sent to browser
  projectId,
  projectEnvironment: 'development',
});

const origin = request.headers.get('origin');

const tokenResponse = await pd.tokens.create({
  externalUserId: String(userId),  // "42"
  allowedOrigins: [origin],        // ["https://app.kuse.ai"]
});

// tokenResponse = {
//   token: "ct_abc123...",
//   expires_at: "2026-03-28T12:30:00Z"  // ~5 min TTL
// }
7
Return Token to Browser 返回 Token 到浏览器
The server responds with the full token response. The Pipedream SDK in the browser receives it via the tokenCallback promise resolution. The SDK now has a valid connect_token to authenticate its iframe.
服务端返回完整的 token 响应。浏览器中的 Pipedream SDK 通过 tokenCallback promise 接收。SDK 现在有了有效的 connect_token 来认证 iframe。
app/api/connectors/token/route.ts (line 68)

8
OAuth Popup / iframe 打开 OAuth 弹窗
The Pipedream SDK opens an iframe/popup containing the OAuth authorization page. The connect_token tells Pipedream which user and project this auth is for. The user sees the third-party provider's consent screen (e.g., "Allow KUSE to access your GitHub repos?"). After user grants access, the provider redirects back to Pipedream with an auth code, which Pipedream exchanges for access/refresh tokens.
Pipedream SDK 打开一个包含 OAuth 授权页面的 iframe/弹窗。connect_token 告诉 Pipedream 这次授权是哪个用户和项目。用户看到第三方提供商的授权页面（如"允许 KUSE 访问你的 GitHub 仓库？"）。用户授权后，提供商带着 auth code 重定向回 Pipedream，Pipedream 用它交换 access/refresh tokens。
next.config.ts (COOP header required)
// next.config.ts — Required for popup to work
async headers() {
  return [{
    source: '/(.*)',
    headers: [{
      key: 'Cross-Origin-Opener-Policy',
      value: 'same-origin-allow-popups'
      // Without this, the OAuth popup gets blocked!
    }]
  }];
}
9
onSuccess Callback 成功回调
After successful OAuth, Pipedream SDK calls your onSuccess callback with the new account ID. Your app stores this in the Zustand connector store for UI updates, then auto-closes the Pipedream iframe via postMessage.
OAuth 成功后，Pipedream SDK 调用你的 onSuccess 回调，带上新的账户 ID。应用将其存入 Zustand connector store 更新 UI，然后通过 postMessage 自动关闭 Pipedream iframe。
components/CustomizeView/ConnectorsTab.tsx (lines 108-120)
await client.connectAccount({
  app: connector.id,
  onSuccess: (res) => {
    // res.id = Pipedream account ID (e.g. "apn_abc123")
    store.setConnected(connector.id, res.id);

    // Invalidate React Query cache to refresh account list
    queryClient.invalidateQueries({ queryKey: ['accounts'] });

    // Auto-close the Pipedream iframe
    window.postMessage({ type: 'close' }, '*');
  },
  onError: (err) => {
    console.error('OAuth failed:', err);
  },
});

Security Considerations 安全注意事项

  • Secret isolation 密钥隔离: PIPEDREAM_CLIENT_SECRET lives only on the server — never in NEXT_PUBLIC_* env vars. The browser only sees the short-lived connect_token.
    PIPEDREAM_CLIENT_SECRET 只存在于服务端 — 不放在 NEXT_PUBLIC_* 环境变量中。浏览器只能看到短时效的 connect_token。
  • User scoping 用户隔离: externalUserId binds the connect_token to one user. If you pass the wrong ID, one user can see another user's connected accounts.
    externalUserId 将 connect_token 绑定到一个用户。如果传错 ID，一个用户可以看到另一个用户的连接账户。
  • Origin restriction 来源限制: allowedOrigins prevents token theft via CORS — only your app's domain can use the token.
    allowedOrigins 通过 CORS 防止 token 被盗 — 只有你的应用域名能使用这个 token。
  • COOP header: Cross-Origin-Opener-Policy: same-origin-allow-popups is required or OAuth popups get silently blocked.
    需要设置 Cross-Origin-Opener-Policy: same-origin-allow-popups，否则 OAuth 弹窗会被静默阻止。
  • Token TTL 令牌过期: connect_token is short-lived (~5 min). SDK auto-refreshes via tokenCallback when needed.
    connect_token 短时效（约5分钟）。SDK 在需要时通过 tokenCallback 自动刷新。
