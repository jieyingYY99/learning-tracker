1 · Direct-Manipulation Geometry 画布直接操作几何 frontend

拖拽移动 · 两阶段点击选择 · 8-handle resize · 边界 clamp · flow→absolute 冻结

What it is 这是什么

deck 画布上直接拖动/缩放绝对定位的 shape。几何计算在 iframe bridge runtime（ES5 字符串脚本，可同源单测），数据模型是 left/top/width/height + anchor 中性化（拖拽时写 right/bottom:auto，防止固定的右/下锚点把宽高拉伸而非移动）。clamp 分两个 regime。
Drag/resize absolutely-positioned shapes directly on the deck canvas. Geometry math lives in the iframe bridge runtime (ES5 string, unit-testable). Model is left/top/width/height + anchor neutralization; clamp has two regimes.

📁 deckDragGeometry.ts · deckSelectionLogic.ts · deckFreezeLogic.ts · deckStageBridgeRuntime.ts
  • FITS vs OVERSIZED clamp：shape 小于 slide → 接触边界即停 [0, section-size]；shape 大于 slide（full-bleed 出血条纹）→ 允许超出但保留 minVisible px 可抓 [minVisible-size, section-minVisible]。
    普通元素和故意出血的背景条用不同夹取策略。
  • detach text from clip：text shape 用 autofit（overflow:hidden），拖拽时临时设 overflow:visible 让被裁的文本可见 → 必须按 visible extent（box∪scrollSize）而非 box 计算夹取。
    这是「拖拽时能看到完整文本」UX 的代价：坐标系变复杂。
  • 三层坐标系：slide 设计空间 px → iframe visual px → host viewport px，scale = iframeRect.width / iframe.offsetWidth 恢复 zoom。这是项目中最复杂的几何交互。
  • flow→absolute 冻结：点「Enable drag editing」时遍历 DOM，把叶子块/有皮肤的容器测量成绝对 shape，重测 drift > 0.5px 则整片回滚（从 cssText 备份恢复）。

function kuseDragClampAxis(pos, size, sectionSize, minVisible) {
  if (size < sectionSize) {
    // FITS：接触边界即停
    return clamp(pos, 0, sectionSize - size);
  }
  // OVERSIZED（full-bleed 出血）：允许超出但保留可抓边缘
  return clamp(pos, minVisible - size, sectionSize - minVisible);
}

2 · Floating Text Toolbar over iframe contentEditable iframe 文本浮动工具栏 frontend

bold/italic · list · link · heading level · typography —— 经 postMessage bridge，不走 execCommand 也不走 Tiptap

What it is 这是什么

选中 shape 后浮现的格式化工具栏。编辑内容是 iframe 内第三方引擎的原生 HTML（不受 ProseMirror schema 管辖），所以命令经 bridge 分层 action：applyStyles（块级）/ applyRangeStyle（span 包裹）/ applyList（重建 p↔ul）/ applyLink。定位用 iframe 上报的 selection rect + 纯函数碰撞避让。
Format toolbar floating over a selected shape. Content is the iframe engine's native HTML (not ProseMirror), so commands go through layered bridge actions, positioned via the reported selection rect.

📁 DeckTextToolbar.tsx · deckTextToolbarLogic.ts · deckToolbarPlacement.ts · deckSelectionRect.ts

3 · Non-Destructive Image Crop 非破坏性图片裁剪 frontend

transform 几何（crop-to-scale）vs clip-path 蒙版（crop-to-shape）—— 两条非破坏路径

What it is 这是什么

crop-to-scale：源分数坐标 CropRect[0..1] 编码为 IMG 的 transform: scale(1/w,1/h) translate(...) + object-fit:fill，缩放与拉伸互相抵消保证无失真。crop-to-shape：clip-path 蒙版加在未变换的 figure 父元素（不是被 transform 的 img，否则圆变椭圆）。
Two non-destructive crop paths: transform geometry (scale-crop, fractional source rect) and clip-path mask (shape-crop, on the untransformed parent).

📁 deckImageCropLogic.ts · deckImageShapeLogic.ts · DeckImageCropEditor.tsx · DeckImageReplacePanel.tsx
  • NO-DISTORTION INVARIANT：rect 宽高比 = figure 宽高比时 scale(1/w,1/h) 才能抵消 fill 拉伸 → 选 16:9 preset 必须同步 pin figure aspect-ratio: 16/9。
  • 蒙版加在 figure 不加在 img：CSS clip-path 在元素 local（pre-transform）坐标系解析再应用 transform —— 加在被 scale 的 img 上圆会变椭圆，加在无变换的 figure 上几何恒定。
  • id marker 往返，不匹配 clip-path：浏览器把 circle(50%) 规范化为 circle(50% at 50% 50%)，字符串匹配失效 → 用 --deck-shape 自定义属性（字面存储不规范化）往返 active 状态。
  • replace 只换 src：swapSrc 不写 style patch，新图自动继承 crop+shape（img node 和 figure parent 不变，分数坐标对任意宽高比有效），保留用户工作流。

4 · Undo/Redo: Snapshot + In-Place Hybrid 撤销/重做：快照+原地补丁混合 frontend

structural op 存整片 HTML（remount 闪烁）· in-place op 存补丁（bridge 原地打补丁不闪烁）

What it is 这是什么

混合两种 op：structural（增删改幻灯片等结构改动）存整片 before/after HTML，restore 时 bump structuralVersion 触发 iframe remount；in-place（样式/文本/图片 src 等非结构改动）存 per-node 补丁，经 iframe:patch 事件 + setNodeHtml/applyStyles/replaceImageSrc 原地打补丁，不 remount、不闪烁、选区不丢。
Hybrid: structural ops store whole-doc HTML (remount), in-place ops store per-node patches replayed via bridge (no flash, selection kept).

📁 deckHistory.ts · DeckPreview.ts · deckPatchRouter.ts · useDeckBridge.ts
核心竞态 bug： 双 undo 时，structural restore 让 iframe 正在重载，紧随的 in-place postMessage 被丢弃（bridge 监听器还没重新挂上）→ host 缓存与 iframe DOM 静默分歧，无自愈。
Race bug: on a double-undo, a structural restore reloads the iframe; the next in-place postMessage is dropped (listener not re-attached) → cache and iframe silently diverge.
  • version-tag stash 修复：in-place 补丁打上当前 structuralVersion 暂存，iframe load 后版本匹配才 replay，不匹配则丢弃为 stale（文档反正已变形）。
  • structuralVersion 在 restore 时 unused：op 记录时存的版本是死值，restore 要新鲜读 _structuralVersion 才有意义。
  • bridge 拒绝的 URL 要 skip 且 no-emit：保持 host 缓存与 iframe DOM lock-step，避免分歧。
  • in-place 选区不丢：node path 不变，无需 remount 后 re-anchor（structural op 才需要）。

6 · Cross-Process Insert: Pending + Post-Remount Auto-Select 跨进程插入：pending + 重挂载后自动选中 frontend

数据驱动工具栏 · 插入 Text/Image/Shape 到 iframe 后异步选中 · 竞态清理

What it is 这是什么

顶部工具栏（数据驱动 placeholder 名册 + icon-only 分组）的插入栏把元素插到 iframe 内第三方 HTML。因为插入要改 host cache → bump structuralVersion → iframe remount（异步）→ 新节点才进 DOM，host 无法同步访问，所以用 pending + consume：记录路径快照，iframe onLoad 时一次性消费并经 bridge 选中。
Insert bar inserts elements into the iframe's HTML. Since insert triggers an async remount, the host can't access the new node synchronously — so it uses pending + post-remount consume.

📁 DeckTopToolbar.tsx · deckTopToolbarItems.ts · DeckPreview.ts · DeckStage.tsx
  • pending path 是绝对索引快照 [slideIndex, childIndex]：任何改 slide/child-index 的 sibling op（delete/duplicate/reorder/imageReplace）都会让它失效 → 必须 _invalidatePendingInsertSelection() 防误选。
  • 一次性消费语义：takePendingInsertSelection() 取出后立即清字段，避免被消费多次。
  • 插入文本故意不加 data-kuse-edit 标记：否则翻转整个 deck 进 whitelist regime，反而让原有无标记段落停止编辑（Heuristic regime 下新文本自动可编辑）。
  • no-op guard：插到 empty/generating slide 返回 null 不记录 pending。

// selection 相对当前 doc，transaction 原子，无竞态窗口
Q: 插入元素后怎么选中？ 同进程可访问 DOM + 插入与选中原子（同 transaction）+ 无长竞态窗口 → 同步选中 跨进程（iframe/worker/remote）+ 插入与重建异步边界 → pending + post-mount consume 同进程异步定位新建元素（如新建文件夹进编辑态） → store event + rAF + setEditingPath 竞态清理必做：所有 sibling structural op 清 pending；消费用 take 非 get；no-op guard；一次性语义

不是「这是什么」，而是「这个场景你会选哪种方案？为什么」。点选项查看答案。

Q1. 你要做一个 Kanban 看板，卡片在列之间拖拽，只需要改变卡片所属的列和顺序，不涉及任意像素坐标。用哪种拖拽方案？A Kanban board where cards drag between columns — only logical position (column + order) changes, no pixel coordinates.
正确：B。 卡片是有序集合、拖拽只改逻辑位置（columnId + index）、无几何约束、不在 iframe —— 这正是 @dnd-kit 的甜区。手写几何（A）是自由坐标画布才需要的重型方案，在这里是过度工程。marquee（C）是范围选择不是拖放排序。
Q2. 你要给一张图片加圆形遮罩做头像，同时这张图片本身用 CSS transform 做了缩放裁剪。clip-path 应该加在哪？Round mask on an image that already has a CSS transform scale-crop. Where to put clip-path?
正确：B。 CSS clip-path 在元素的 local（pre-transform）坐标系解析再应用 transform。加在被 scale(1/w,1/h) 的 img 上，圆会被同样拉伸成椭圆（A 错）。反向计算（C）复杂且脆弱。加在无变换的 figure 父元素上，蒙版几何与 crop transform 正交、恒定 —— 这正是 deck crop-to-shape 的设计。
Q3. 你的编辑器做撤销：内容是一个大文档，用户每秒打很多字，你想要撤销时不闪烁、保留光标，且改动可以经编辑器的命令通道重放。用哪种 undo 范式？Frequent fine-grained edits, want no-flash undo that keeps the cursor and can replay through the editor's command channel.
正确：B。 频繁细粒度编辑 + 有原地打补丁通道 + 要避免 remount 闪烁/保选区 + op 可经同一 mutator 重放 → op-based 原地补丁（deck 非结构改动正是这样）。纯快照（A）restore 必整片重排（闪烁、丢光标）。backup-restore（C）适合粗粒度文件操作。注意：如果是跨进程异步 remount，还要加 version-tag 暂存防竞态。
Q4. 你给一个只有 3 个事件、字段各不相同、互相无关联的功能加埋点。要不要建集中 typed wrapper + session + funnel？3 events, all different fields, unrelated. Build centralized wrapper + session + funnel?
正确：B。 选对抽象等级是关键，不是越复杂越好。event <5 + 无共享 common + 不需聚合 + 无竞态 + 不成 funnel → 散点直埋足够（项目里 dispute/scheduled-tasks 就是这样，没问题）。集中 typed wrapper（A）在 event≥10 + 高 common 重复 + funnel 需求时才值得。session（C）只在「一次任务含多子操作 + 要统计会话级转化」时引入。
Q5. 你在 iframe 里插入一个元素，插入会触发 iframe 重挂载，之后你想选中这个新元素。但插入后、重挂载完成前，用户可能删了别的元素。怎么保证不误选？Insert into iframe triggers remount; you want to select the new element after; but user may delete other elements before remount completes.
正确：B。 跨进程异步边界下，host 无法同步访问 iframe 新节点（A 错，selectShape 此时新节点还没进 DOM）。pending path 是绝对索引快照 [slideIndex, childIndex]，任何改 slide/child-index 的 sibling op（delete/duplicate/reorder）都会让它失效，所以必须在每个这类 op 里清除 pending，并在 onLoad 后用 take（一次性）消费。setTimeout（C）是脆弱的时序赌博，无法防竞态。
Q6. 你的富文本工具栏，用户 drill into 一个加粗的 span，然后点「居中对齐」。对齐应该作用在哪个元素上？User drills into a bold span, clicks "center align". Which element should it apply to?
正确：B。 text-align / indent / heading level 是块级属性，作用在 inline span 上无效（padding-left 在 span 上不起作用）。所以块级操作必须 resolveBlockTargetPath(path, outerPath) 回到外层块。deck 还把 resolveIndentScope() 硬编码返回 'block'。range 级属性（color/font）才作用在 span。

Anchor Neutralization
deck 画布拖拽 / drag
拖拽绝对定位 shape 时写 right/bottom: auto，防止固定的右/下锚点把宽高拉伸而非移动。PPTX 绝对定位可能有 right/bottom anchor，不中性化会导致拖拽变拉伸。
FITS vs OVERSIZED clamp
边界夹取 / containment
两个夹取 regime：shape < slide → FITS（接触边界即停）；shape ≥ slide（full-bleed 出血）→ OVERSIZED（允许超出但保留 minVisible px 可抓）。单一 clamp 会破坏出血设计意图。
Visible Extent
detach text / clamp
max(boxSize, scrollSize)。text shape 拖拽时 detach overflow 让 autofit 文本可见，因此夹取要按 box ∪ 溢出内容的总范围，而非仅 box 尺寸。
Three Coordinate Systems
deck 坐标 / coords
slide 设计空间 px → iframe visual px → host viewport px。scale = iframeRect.width / iframe.offsetWidth 恢复 zoom。项目中最复杂的几何交互。
Block-Scope vs Range-Scope
文本工具栏 / toolbar
块级（level/align/indent）→ 强制回外层块 resolveBlockTargetPath(path, outerPath)，drill-in span 上无效。range 级（color/font）→ 作用在 selection span。
NO-DISTORTION INVARIANT
图片裁剪 / crop
crop-to-scale 无失真条件：rect 宽高比 = figure 宽高比，此时 scale(1/w,1/h) 才能抵消 object-fit:fill 的拉伸。选比例 preset 必须同步 pin figure aspect-ratio。
clip-path on Untransformed Parent
形状裁剪 / shape-crop
clip-path 加在未变换的 figure 父元素，不加在被 transform 的 img。CSS 在 local（pre-transform）坐标系解析 clip-path 再应用 transform —— 加在 img 上圆会变椭圆。
ID Marker Round-Trip
active 状态 / round-trip
用 --deck-shape 自定义属性往返 active 形状，不匹配 clip-path 字符串（浏览器把 circle(50%) 规范化为 circle(50% at 50% 50%)，字符串匹配失效）。自定义属性字面存储不规范化。
Structural vs In-Place Op
undo 混合 / hybrid
structural（增删幻灯片）存整片 HTML，restore 触发 iframe remount（闪烁）；in-place（样式/文本/src）存 per-node 补丁，经 bridge 原地打补丁（不闪烁、保选区）。
Version-Tag Stash
in-flight remount 竞态
in-place 补丁打上当前 structuralVersion 暂存，iframe load 后版本匹配才 replay，不匹配则丢弃为 stale。解决「structural restore 让 iframe 重载时 in-place postMessage 被丢弃」的静默分歧。
Gate on Real Mutation
埋点准确性 / accuracy
mutation 函数返回 bool，埋点门控在真实 mutation：if (applyImageReplace(...)) track(...)。no-op（URL 相同）不发事件，避免幽灵事件污染数据。
Session Roll-Up
埋点 session / analytics
打开建 session（UUID + counter Set + did* flags），关闭发 session_ended 含聚合指标（slides_edited、ai_slides_created、did_export）。GA4 可 JOIN 同会话事件，一个 event 给所有会话级指标。
Pending + Post-Remount Consume
跨进程插入 / insert
跨进程插入元素后选中：记录 pending 路径快照 + bump 触发 remount，iframe onLoad 后用 take 一次性消费。每个 sibling structural op 必须清除 pending 防路径失效误选。
Whitelist vs Heuristic Regime
deck 可编辑性 / editability
deck 有 data-kuse-edit 标记 → whitelist（只标记节点可编辑）；无标记 → heuristic（自动分类可编辑）。插入文本故意不加标记，否则翻转整个 deck 进 whitelist 让原有段落停止编辑。
