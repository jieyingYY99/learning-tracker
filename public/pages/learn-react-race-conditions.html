React Race Conditions: A Systematic Mental Model

React 竞态条件：系统化心智模型 — 基于 Kuse App V2 中修复的 4 个真实 bug 和 5 种防护模式

Why this matters 为什么重要

This week you fixed 4 race condition bugs using 5 different patterns, but the patterns were discovered ad-hoc during debugging. You need a systematic mental model for when React re-renders cause async overlap.

本周你用 5 种不同模式修复了 4 个竞态 bug，但模式是调试过程中临时发现的。你需要一个系统化的心智模型来理解 React 重渲染何时导致异步重叠。

The Core Problem 核心问题

React re-renders are synchronous, but the operations they trigger are often asynchronous. When a re-render fires while an async operation from a previous render is still in flight, you get a race condition.

React 重渲染是同步的，但它们触发的操作往往是异步的。当一个重渲染在上一次渲染的异步操作还在执行中时触发，就会产生竞态条件。

// The universal race condition template 竞态条件通用模板
useEffect(() => {
  // Render N: start async work
  fetchSomething().then(result => {
    // Render N+1 may have already happened!
    // This result is now STALE but still wants to setState
    setState(result); // Bug: overwrites newer data
  });
}, [dependency]); // dependency changes trigger Render N+1

5 Patterns Deep Dive 五种模式深度分析

Each pattern with real code, trigger scenario, and gotchas from the Kuse App V2 codebase
每种模式配有真实代码、触发场景和来自 Kuse App V2 代码库的注意事项

1
Version Counter 版本计数器
Invalidate stale async results without canceling them 作废过时异步结果而不取消它们
STALE CLOSURE useConversationManager.ts 
// Version counter: incremented by createConversation so that any
// in-flight loadConversationMessages can detect staleness.
const loadVersionRef = useRef(0);

async function loadConversationMessages() {
  const v = loadVersionRef.current;  // Capture version BEFORE await

  const messages = await fetchMessages(conversationId);

  if (v !== loadVersionRef.current) {  // Check AFTER await
    setState(prev => ({ ...prev, isLoadingMessages: false }));
    return;  // Discard stale result 丢弃过时结果
  }
  // ... apply messages ...
}

async function createConversation() {
  loadVersionRef.current++;  // Invalidate ALL in-flight loads
  // ... create new conversation ...
}
Trigger Scenario 触发场景
  1. User switches view → ChatInterface remounts
  2. Init effect fires loadConversationMessages() (async, takes ~200ms)
  3. Simultaneously, handleProcessSendToAgent calls createConversation()
  4. Without fix: stale history from step 2 overwrites new conversation messages
  5. With fix: version counter mismatch → stale load discarded
Deep Dive: When to use 深度分析：何时使用

Use when 使用场景： You have an async operation (fetch, computation) whose result should only be applied if the "context" hasn't changed while it was running.

Key insight 关键洞察： The stale request still completes (wastes bandwidth), but its result is safely discarded. This is simpler than AbortController when you don't need to cancel the network request itself.

Gotcha 注意事项： Early-return paths (validation failures, empty state checks) must still reset loading states, otherwise UI gets stuck in loading.

vs AbortController： Version counter lets the request finish but ignores it. AbortController cancels it mid-flight. Use AbortController when the response is large (streaming) or the server does expensive work you want to stop.

2
Dual Guard 双重守卫
State + Ref dedup to prevent effect re-fire 状态 + Ref 去重防止 Effect 重复触发
EFFECT RE-FIRE ChatInterface/index.tsx 
const processedPayloadRef = useRef<string | null>(null);

useEffect(() => {
  if (!sendToAgentPayload) return;  // Guard 1: state null check

  const payloadKey = JSON.stringify(sendToAgentPayload);
  if (payloadKey === processedPayloadRef.current) return;  // Guard 2: ref dedup
  processedPayloadRef.current = payloadKey;

  try {
    await sendMessage(sendToAgentPayload);
    sendToAgent(null);            // Clear STATE
    processedPayloadRef.current = null; // Clear REF
  } catch (error) {
    sendToAgent(null);            // CRITICAL: clear state on error too!
    processedPayloadRef.current = null;
  }
}, [sendToAgentPayload]);
Trigger Scenario 触发场景
  1. User sends message → sendToAgentPayload state is set
  2. Effect runs, processes payload, calls API
  3. API throws error mid-processing
  4. Old code: only clears ref, not state → React's batched setState still holds stale payload → effect re-fires → duplicate message sent
  5. Fix: clear BOTH state AND ref in error catch → no re-fire
Deep Dive: Why two guards? 深度分析：为什么需要两层守卫？

Guard 1 (state null check)： Prevents effect from running when there's nothing to process. Simple and fast.

Guard 2 (ref JSON dedup)： React may batch multiple setState calls, causing the effect to re-run with the same payload. The ref ensures identical payloads are never processed twice.

Why both are needed 为什么两者都需要： State alone can't prevent batching duplicates. Ref alone can't prevent null-state triggers. Together they cover both React-scheduling races and user-action races.

Gotcha 注意事项： JSON.stringify has performance cost for large payloads. For simple primitives, direct comparison is better. Only use JSON.stringify when the payload is an object.

3
Boolean Flag 布尔标志
Mutex-like guard for async operations 类互斥锁的异步操作守卫
CONCURRENT EXECUTION ConnectorsTab.tsx 
const connectingRef = useRef(false);

const handleDirectConnect = useCallback(async (connector) => {
  if (connectingRef.current) return;  // Reject if already connecting
  connectingRef.current = true;

  try {
    await pipedreamConnect({
      onSuccess: () => { /* handle success */ },
      onError:   () => { connectingRef.current = false; },
      onClose:   () => { connectingRef.current = false; },
    });
  } catch {
    connectingRef.current = false;  // Reset in ALL exit paths!
  }
}, []);
Trigger Scenario 触发场景
  1. User clicks "Connect" for Slack → OAuth popup opens, connectingRef = true
  2. User rapidly clicks "Connect" for Zapier before Slack finishes
  3. Without guard: two OAuth flows start → responses arrive out-of-order → confused state
  4. With guard: second click blocked until connectingRef = false
Deep Dive: Ref vs State for guards 深度分析：Ref vs State 作为守卫

Why useRef, not useState 为什么用 useRef 而不是 useState：

  • Refs don't trigger re-renders → safe to skip dependency arrays
  • State changes are batched and asynchronous → a rapidly-clicked button might see stale state
  • Ref updates are synchronous → guaranteed to block the next click immediately

Critical rule 关键规则： You MUST reset the flag in every exit path: onSuccess, onError, onClose, AND catch. Missing one path = permanent lock.

4
AbortController 中断控制器
Cancel in-flight streams on context change 上下文变化时取消进行中的流
UNCANCELLED STREAM useAIStream.ts 
const abortControllerRef = useRef<AbortController | null>(null);

// When starting a stream 开始流时
abortControllerRef.current = new AbortController();
fetch(url, { signal: abortControllerRef.current.signal });

// Inside the processing loop 处理循环内
for await (const event of eventStream) {
  if (abortControllerRef.current?.signal.aborted) break;
  // process event...
}

// User clicks "Stop" 用户点击"停止"
function stopStream() {
  abortControllerRef.current?.abort();
}

// Cleanup 清理
finally { abortControllerRef.current = null; }
Trigger Scenario 触发场景
  1. User sends message → stream starts processing 50 SSE events
  2. After 5 events, user clicks "Stop"
  3. stopStream() calls abort()
  4. Without abort check in loop: continues rendering all 50 events
  5. With check: loop breaks at event #6, UI stops updating
Deep Dive: AbortController lifecycle 深度分析：AbortController 生命周期

Must create NEW per request 每次请求必须新建： An aborted AbortController stays aborted forever. Reusing it would abort the next request immediately.

Signal check placement 信号检查位置： Check inside the loop (passive cancellation), not before the loop. This ensures cleanup logic in the loop body still runs for the current event.

vs Version Counter： AbortController cancels the network request itself (saves bandwidth for large streams). Version counter only discards the result (request still completes). For SSE streams with 50+ events, AbortController is clearly better.

5
Timer Suppression 定时器抑制
Re-arming timer to suppress post-interaction events 重新激活定时器抑制交互后事件
EVENT TIMING useListDragSelect.ts 
const suppressClickRef = useRef(false);
const suppressTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);

function armSuppressClick(ms: number) {
  suppressClickRef.current = true;
  if (suppressTimerRef.current) clearTimeout(suppressTimerRef.current);
  suppressTimerRef.current = setTimeout(() => {
    suppressClickRef.current = false;
  }, ms);
}

// Drag starts: arm with 200ms
armSuppressClick(200);

// Drag ends (commitAndReset): RE-ARM with 300ms!
armSuppressClick(300);  // This is the critical fix

// Click handler checks and consumes
if (suppressClickRef.current) {
  suppressClickRef.current = false;
  clearTimeout(suppressTimerRef.current);
  return true;  // Suppress click
}
Trigger Scenario 触发场景
  1. User starts drag at 0ms → timer armed for 200ms
  2. User drags slowly for 300ms (normal behavior)
  3. At 200ms: initial timer fires → suppressClickRef = false
  4. At 300ms: user releases → pointerup fires
  5. Browser generates click event at ~305ms
  6. Without re-arm: flag is already false, click proceeds → clears selection!
  7. With re-arm: commitAndReset() calls armSuppressClick(300) → flag is true at 305ms → click suppressed
Deep Dive: Browser event ordering 深度分析：浏览器事件顺序

The problem 问题： Browser always fires pointerdown → pointermove → pointerup → click in this order. The click event is automatically generated by the OS after pointerup. You can't prevent it in pointerup handler.

Why re-arm is needed 为什么需要重新激活： The initial timer (200ms) is just a heuristic for "we started dragging." But if the drag lasts longer than 200ms, the timer expires before the drag ends. The re-arm at commitAndReset creates a fresh window after the drag completes.

Critical invariant 关键不变量： SUPPRESS_CLICK_DRAG_END_MS (300ms) MUST be >= SUPPRESS_CLICK_DRAG_START_MS (200ms). If end < start, there's a window where the click can sneak through.

Decision Flowchart 决策流程图

When you encounter async behavior in React, use this flowchart to pick the right pattern
当你在 React 中遇到异步行为时，使用此流程图选择正确的模式

Q1: Is the race between async operations or DOM events?
问：竞态是在异步操作之间还是 DOM 事件之间？

├─ DOM events (click after drag, event ordering)
│ DOM 事件（拖拽后的点击、事件顺序）
│ └─ → Pattern 5: Timer Suppression 定时器抑制

└─ Async operations (fetch, streams, OAuth)
异步操作（请求、流、OAuth）

Q2: Do you need to CANCEL the operation, or just IGNORE the result?
问：你需要取消操作，还是只需忽略结果？

├─ CANCEL (streaming, large payloads, server-side work)
│ 取消（流式、大负载、服务端工作）
│ └─ → Pattern 4: AbortController 中断控制器

└─ IGNORE result (lightweight fetch, won't waste much)
忽略结果（轻量请求、浪费不大）

Q3: Is the race from the SAME effect re-firing, or DIFFERENT operations overlapping?
问：竞态来自同一 Effect 重复触发，还是不同操作重叠？

├─ Same effect re-firing (React batched setState, strict mode)
│ 同一 Effect 重复触发（React 批量 setState、严格模式）
│ └─ → Pattern 2: Dual Guard 双重守卫

└─ Different operations overlapping
不同操作重叠

Q4: Should the second call be BLOCKED, or just make the first one STALE?
问：第二次调用应该被阻止，还是只让第一次变成过时？

├─ BLOCK (OAuth popups, one-at-a-time operations)
│ 阻止（OAuth 弹窗、一次一个的操作）
│ └─ → Pattern 3: Boolean Flag 布尔标志

└─ First result becomes STALE (load history → create new chat)
第一次结果变为过时（加载历史 → 创建新聊天）
└─ → Pattern 1: Version Counter 版本计数器
