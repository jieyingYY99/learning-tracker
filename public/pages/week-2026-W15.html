SSE Event-Driven File Tree Refresh SSE 事件驱动文件树刷新

How SSE deliverables and turn_end events flow through streamProcessor → mutationAdapter → refreshScheduler to update the file tree. Combines immediate mutations with debounced batch refreshes.

SSE 事件如何通过三层管线触发文件树增量更新：即时 mutation + 延迟批量防抖刷新。

Data Flow Pipeline 数据流管线

SSE Event (deliverables/turn_end) ↓ streamProcessor.ts — extract files, normalize to ExistingFileInfo ↓ mutationAdapter.ts — immediate: nodesByPath[path] = node; treeVersion++ ↓ refreshScheduler.ts — debounced: 800ms per-folder, seq dedup ↓ listDir() — batch refresh folder listing from API

Three Side-Effect Patterns 三种副作用模式

Pattern 模式When 何时使用Latency 延迟
Direct mutation 直接变更File create/delete (must be instant) 文件增删0ms
Debounced callback 防抖回调Skills refresh, folder listing 技能刷新800ms
Event bus 事件总线UI state (panels, selections) UI 状态~0ms
Decision Guide 决策指南
  • Must be visible immediately 必须即时可见 → Direct store mutation
  • Can tolerate 800ms delay 可容忍延迟 → refreshScheduler debounce
  • Third-party list refresh 第三方列表 → setTimeout debounce
  • Delete: ONLY mutation, NO parent refresh 删除：仅 mutation，不刷新父目录（避免已删文件闪现）

RefreshScheduler: Sequence Dedup 序号去重

// Per-folder debounce with stale-check 每文件夹防抖 + 过期检查 schedule(path) { const mySeq = ++seq[path]; clearTimeout(timers[path]); timers[path] = setTimeout(() => { if (seq[path] === mySeq) listDir(path); // Only run if no newer event }, 800); }

Cron Schedule & Polling Architecture Cron 调度与轮询架构

Bidirectional cron expression ↔ UI conversion supporting 6 frequency types (every-N-minutes, hourly, daily, weekly, monthly, once). Multi-layer adaptive polling that adjusts interval based on task running state.

Cron 表达式双向转换 + 多层自适应轮询（根据运行状态动态调整间隔）+ optimistic run 插入。

Cron ↔ UI Conversion 双向转换

// parseCron(): string → UI state "*/30 * * * *" → { frequency: 'every_minutes', intervalMinutes: 30 } "0 * * * *" → { frequency: 'hourly', minute: 0 } "0 9/2 * * *" → { frequency: 'hourly', hour: 9, intervalHours: 2 } "30 14 * * 1" → { frequency: 'weekly', minute: 30, hour: 14, weekday: 1 } "0 8 15 * *" → { frequency: 'monthly', minute: 0, hour: 8, dayOfMonth: 15 } "0 10 25 4 *" → { frequency: 'once', with calendar date picker } // buildCron(): UI state → string (reverse of above)

Quiz 测验

Test your understanding of this week's concepts
测试你对本周概念的理解

1. You're building a "cancel membership" feature where cancellation takes effect at the end of the billing period. Which pattern should you use?
你在构建一个"取消会员"功能，取消在账单周期结束时生效。应该使用哪种模式？
Deferred + cancellable transitions involving money should use a state machine pattern. The server determines effective_at and the user can restore/cancel before that date. Never use optimistic updates for financial operations.
涉及金钱的延迟可取消转换应使用状态机模式。服务端确定生效时间，用户可在生效前取消。金融操作绝不使用乐观更新。
2. An AI agent creates 5 files rapidly via SSE. How does the file tree handle this without excessive API calls?
AI agent 通过 SSE 快速创建了 5 个文件。文件树如何在不过度调用 API 的情况下处理？
The hybrid approach: each file gets an immediate store mutation (nodesByPath update) so it appears instantly. The refreshScheduler debounces folder listing to 800ms, so all 5 files result in just one listDir() call. The seq dedup ensures stale refreshes don't execute.
混合方案：每个文件即时 store mutation 让它立即显示。refreshScheduler 将文件夹列表防抖到 800ms，5 个文件只触发一次 listDir()。seq 去重确保过期刷新不执行。
3. A scheduled task runs every 2 hours starting at 9 AM. What cron expression does buildCron() generate?
一个定时任务从早上 9 点开始每 2 小时运行一次。buildCron() 生成什么 cron 表达式？
0 9/2 * * * means "at minute 0, starting at hour 9 and every 2 hours after." This is the standard cron step notation for hourly intervals with a start offset. */2 would start from 0, not 9.
0 9/2 * * * 表示"在第 0 分钟，从第 9 小时开始每 2 小时"。这是带偏移的标准 cron 步进表示法。*/2 从 0 开始而非 9。
4. You upload a skill file and the server returns HTTP 409. What happens in the UploadSkillDialog?
上传技能文件时服务端返回 HTTP 409。UploadSkillDialog 中会发生什么？
409 means the skill already exists. The dialog shows a confirmation asking if the user wants to replace it. If confirmed, the upload is retried with force=true. The forceReplace prop from the SkillCard menu skips this confirmation entirely.
409 表示技能已存在。弹窗询问用户是否替换。确认后以 force=true 重新上传。从 SkillCard 菜单的 forceReplace prop 会跳过此确认。
5. The TaskList polls at 30s intervals. A user clicks "Run Now" and the task starts running. What changes?
TaskList 以 30s 间隔轮询。用户点击"立即运行"后任务开始运行。有什么变化？
Two things happen: (1) An optimistic run with run_id: "optimistic-{timestamp}" is inserted at the head of the runs list for immediate feedback. (2) The presence of a "running" task causes the interval to switch from 30s to 10s for faster status updates.
两件事：(1) 插入 optimistic run 提供即时反馈；(2) 有"running"任务导致轮询从 30s 切换到 10s。

Scheduled State Machine

Deferred state transition pattern

延迟调度状态机：变更在未来时间点生效（如账单周期结束），期间可取消（restore）。关键：涉及金钱时绝不乐观更新，全部等服务端确认 scheduled_change === true。

PendingPaymentKey Lock

Scoped double-submit prevention

支付锁模式：用作用域 key（如 personal:pro:monthly）锁定 UI 防止双重提交。比全局 loading 更精确——只锁定对应的按钮，其他 plan 卡片仍可交互。

RefreshScheduler
Sequence Dedup

Per-folder debounce + stale check

防抖刷新调度器：每个文件夹维护 seq 序号，每次 schedule() 递增。800ms 后检查 seq 是否匹配——如果有更新的事件，旧的刷新被自动丢弃。避免 N 个文件创建触发 N 次 listDir()。

Mutation + Refresh
Hybrid Pattern

Immediate visibility + batched API calls

混合副作用模式：文件增删即时写入 nodesByPath（0ms 可见），文件夹列表延迟 800ms 批量刷新。Delete 特殊：只做 mutation 不刷新父目录，防止已删文件闪现。

Adaptive Polling

State-dependent interval switching

自适应轮询：根据数据状态动态调整间隔。TaskList：running → 10s, idle → 30s。TaskDetail：running → 5s, else → 15s。配合 change detection 跳过无变化的重渲染。比 SSE 简单，比固定轮询省带宽。

Cron Step Notation

start/step in cron expressions

Cron 步进表示法：9/2 = 从 9 开始每 2 步（9,11,13,15...）。*/30 = 从 0 开始每 30 步。parseCron ↔ buildCron 双向转换，支持 6 种频率类型。once 模式需要 disabled-time 校验（禁选过去时间）。

409 Conflict Detection

Server-driven conflict resolution

409 冲突检测模式：服务端判断名称冲突比前端预检查更可靠（无竞态），但多一次 round-trip。流程：upload(force=false) → 409 → confirm → upload(force=true)。forceReplace prop 可跳过确认步骤。

Optimistic Run Insert

Synthetic run_id for instant feedback

乐观运行插入：点击"立即运行"时生成合成 run（run_id: "optimistic-{timestamp}"），插入列表头部。服务端返回真实 run 后，通过 run_id 前缀匹配去重替换。提供即时反馈同时保持数据一致性。
