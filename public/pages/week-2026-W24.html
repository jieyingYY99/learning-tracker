1. Per-user vs Device-global Client Storage

客户端存储的数据归属：账号级 vs 设备级
generaldata ownershiplocalStorage

What & Why 是什么 / 为什么

Pinned project ids were first stored under one device-global key kuse_pinned_project_ids. On a shared browser profile, one account's pins leaked to the next user. The fix: key storage by user id (kuse_pinned_project_ids_<userId>), resync on account change, and one-time migrate the legacy global key.

置顶 id 最初存在一个设备级全局 key 下。在共享浏览器配置上，一个账号的置顶会泄漏给下一个用户。修复：按用户 id 分键，账号切换时重新同步，并把旧的全局 key 一次性迁移过去。

The core distinction 核心区分：is this datum a property of the device/browser, or of the account?

  • Device-global 设备级：pure UI prefs that don't belong to anyone — e.g. the section's isExpanded toggle. 纯 UI 偏好，不属于任何账号，如 isExpanded 展开状态。
  • Account-scoped 账号级：anything that references server entities (folder ids, pins, version preference). 任何引用服务端实体的数据（文件夹 id、置顶、版本偏好）。
Deep Dive 深度分析 — code comparison

This week's implementation 本次实现 · store/pinnedProjects/index.ts

const STORAGE_PREFIX = 'kuse_pinned_project_ids_';
const LEGACY_STORAGE_KEY = 'kuse_pinned_project_ids'; // pre-scoping

function getStorageKey(userId) {
  return userId != null ? `${STORAGE_PREFIX}${userId}` : `${STORAGE_PREFIX}guest`;
}

// activeStorageKey follows the signed-in user
function syncPinsToUser(userId) {
  const key = getStorageKey(userId);
  if (key === activeStorageKey) return;
  activeStorageKey = key;
  // first sync to a real user → adopt legacy global pins, then delete them
  if (userId != null && getLocalStorageItem(key) == null) {
    const legacy = sanitizeIds(getLocalStorageItem(LEGACY_STORAGE_KEY));
    if (legacy.length) { savePinnedIds(legacy); removeLocalStorageItem(LEGACY_STORAGE_KEY); }
  }
  usePinnedProjectsStore.setState({ pinnedIds: loadPinnedIds(activeStorageKey) });
}

// subscribe to the user store so pins re-sync on sign-in / switch / logout
useUserStore.subscribe((state) => {
  const userId = state.user?.id ?? null;
  if (userId === lastUserId) return;
  lastUserId = userId; syncPinsToUser(userId);
});

Decision Guide 决策指南

Does the value reference a server entity (id/account)?
  ├─ yes → key by userId + migrate legacy + resync on account change
  └─ no → device-global key is fine (pure UI toggle)

Does it drive a live-rendered list/UI?
  ├─ yes → mirror into a store + push on user change (avoid stale UI)
  └─ no → pull on demand from storage is enough

2. Surviving Component Remounts: Three Cache Strategies

组件重挂载存活：三种缓存策略
frontendRadix Popovercaching

What & Why 是什么 / 为什么

Radix PopoverContent unmounts on close. So the collapsed-rail "My workflows" popover remounts MyWorkflowsList on every open — which originally re-fired listProjects + buildWorkflowAPI.list + fetchWorkflows every single time. The fix used three different caches for three different data sources, in one file.

Radix 弹窗内容关闭即卸载，所以收起态 "My workflows" 弹窗每次打开都会重挂载组件、重新发三个请求。修复方案在一个文件里用了针对三种数据源的三种不同缓存。

Deep Dive 深度分析 — three caches side by side

① Module-level cache + session flag (for MA workflows, fetched via raw API):

// module scope — survives remounts, dies on full page reload
let cachedMAWorkflowProjects = null;
let cachedMAWorkflowDefs = null;
let hasFetchedTurboWorkflowsOnce = false;

useEffect(() => {
  // popover remounts on every open → skip refetch if cache exists
  if (variant === 'popover' && cachedMAWorkflowProjects !== null) return;
  // ... fetch, then: cachedMAWorkflowProjects = merged;
}, [variant]);

② Zustand store cache (for turbo workflows — the store already persists across remounts):

const fetchWorkflows = useWorkflowStore((s) => s.fetchWorkflows);
useEffect(() => {
  if (variant === 'popover' && hasFetchedTurboWorkflowsOnce) {
    setIsInitialTurboReady(true);
    return; // store already holds the list
  }
  void fetchWorkflows().finally(() => { hasFetchedTurboWorkflowsOnce = true; });
}, [fetchWorkflows, variant]);

③ Store-event invalidation (keeps the module cache fresh without re-fetching):

storeEvents.subscribe(STORE_EVENTS.MA_WORKFLOW_CHANGED, ({ siteProjectId, workflowId, workflowName }) => {
  setSiteProjects((prev) => {
    const next = prev.map((sp) => sp.id === siteProjectId ? { ...sp, workflow_id: workflowId } : sp);
    cachedMAWorkflowProjects = next;   // patch the module cache in place
    return next;
  });
});
Strategy 策略Lifetime 生命周期Why here 为何选它
① Module cache + flagUntil page reload 直到刷新页面Data comes from a raw API call with no store of its own — module scope is the cheapest place that outlives a remount. 数据来自无 store 的原始 API，模块作用域是最便宜的「比重挂载活得久」的容器。
② Zustand storeUntil page reload 直到刷新Turbo workflows already live in a store — reuse it, only guard against the redundant per-open fetch. 复用已有 store，只挡掉每次打开的冗余请求。
③ Store eventsLive 实时Cache freshness without polling — chat stream tells us when a workflow changed, so we patch in place instead of re-fetching. 不靠轮询保新鲜——chat 流告诉我们何时变化，原地打补丁。

The key insight 关键洞见: "cache to survive remount" and "keep cache fresh" are two separate problems. ①② solve survival; ③ solves freshness. Conflating them leads to either stale data or wasteful re-fetches.

Decision Guide 决策指南

Does the data already live in a store?
  ├─ yes → reuse store + a session flag to skip per-open refetch
  └─ no → module-level cache (cheapest container that outlives remount)

Can the cache go stale while open elsewhere?
  ├─ yes → add event-driven in-place patching (no re-fetch)
  └─ no → survival cache alone is enough

3. Disciplined Shared-Selector Extraction

有纪律的共享选择器抽取
frontendDRYrefactor

What & Why 是什么 / 为什么

PinnedSection and ProjectsPopover each had the same ~18-line "resolve pinned ids in pin order + remainder" loop. They were merged into one generic splitByPinned(items, pinnedIds, getId). Crucially, AssetsTree's pinned check was left alone — it only needs a boolean includes() membership test, a different shape that wouldn't benefit.

两个组件各有一段约 18 行相同的「按置顶顺序解析 + 余项」循环，合并为通用的 splitByPinned。关键是：AssetsTree 的置顶判断没有动——它只需要布尔 includes() 成员判断，形状不同，抽取无收益。

This is the project's own CLAUDE.md rule in action: "If the same business rule appears in 2+ places, move it to a shared selector. Do NOT extract just to satisfy a line count."

Deep Dive 深度分析 — what was shared vs. what was kept

Extracted 抽取出来的 · store/pinnedProjects/index.ts:

export function splitByPinned(items, pinnedIds, getId) {
  if (pinnedIds.length === 0) return { pinned: [], unpinned: [...items] };
  const pinnedById = new Map();
  const unpinned = [];
  for (const item of items) {
    const id = getId(item);
    if (id != null && pinnedIds.includes(id)) pinnedById.set(id, item);
    else unpinned.push(item);
  }
  // preserve PIN ORDER, not item order — this is the shared business rule
  const pinned = pinnedIds.map((id) => pinnedById.get(id)).filter((x) => x !== undefined);
  return { pinned, unpinned };
}

Used identically by both consumers via a getId adapter:

// ProjectsPopover.tsx
const { pinned, unpinned } = splitByPinned(filteredFolders, pinnedIds, getNodeNumericId);
// PinnedSection.tsx — same call, different item shape

NOT extracted 没有抽取的 · AssetsTree keeps:

// just "is this node pinned?" — a membership test, not an ordered split
pinnedIds.includes(getNodeNumericId(node))
Consumer 使用方Need 需求Decision 决策
PinnedSectionordered pinned group + remainder→ shared splitByPinned
ProjectsPopoverordered pinned group + remainder→ shared splitByPinned
AssetsTreeboolean "is pinned?" per node→ keep inline includes()

The discriminator is "do they share the same business rule, or just the same word?" Both popover/section share pin-order identity resolution (a rule). AssetsTree shares only the concept "pinned" — its logic is genuinely different and forcing it into the shared selector would add a worse abstraction.

Decision Guide 决策指南

Do 2+ call sites share the same business RULE (not just a keyword)?
  ├─ yes → extract a generic selector with an adapter param (getId)
  └─ no → keep it inline — a forced shared abstraction is worse than dupe

Tempted to extract only to hit a line-count target?
  └─ don't. Line count is not a reason to extract (project rule).

4. Version-Counter Signal for Derived Store Reads

版本计数器信号驱动派生读取
frontendZustandmemo

What & Why 是什么 / 为什么

ProjectsPopover needs the list of root folders. Instead of subscribing to a derived array (which would produce a new reference every render and defeat memoization), it subscribes to a single integer treeVersion and reads getState().getRootFolders() inside a useMemo keyed on that counter.

ProjectsPopover 需要根文件夹列表。它没有订阅派生数组（每次渲染产生新引用，破坏 memo），而是订阅一个整数 treeVersion，在以该计数器为 key 的 useMemo 内读取 getState().getRootFolders()。

Deep Dive 深度分析 — counter signal vs. subscribing the array

This week 本次 · ProjectsPopover.tsx:

// subscribe to a stable integer that bumps when the tree changes
const treeVersion = useAssetsStore((s) => s.treeVersion);
const rootFolders = React.useMemo(
  () => useAssetsStore.getState().getRootFolders(), // read derived data imperatively
  [treeVersion]                                      // recompute only on real change
);

The store side 存储侧 · store/assets/index.ts bumps the counter only when it matters:

treeVersion: 0,
// on create/delete/rename:
set((s) => ({ ...s, treeVersion: s.treeVersion + 1 }));
// but for a metadata-only update, bump ONLY if sorting fields changed:
...(bumpVersion ? { treeVersion: s.treeVersion + 1 } : {})

The naive alternative 朴素替代 (what this avoids):

// ❌ getRootFolders() returns a NEW array each call → new reference every render
const rootFolders = useAssetsStore((s) => s.getRootFolders());
// selector returns unstable ref → component re-renders even when nothing changed
// (Zustand compares by Object.is; a fresh array is never === the previous one)
Approach 方案Re-render trigger 触发重渲染Cost 代价
Subscribe derived arrayEvery store change (new ref each time) 每次 store 变化Over-renders; needs a custom equality fn to tame. 过度渲染，需自定义 equality。
Counter signal + getState() (this week)Only when treeVersion bumps 仅版本号变化时Recompute exactly when the tree truly changed. 仅在树真正变化时重算。

The extra subtlety: the store is selective about bumping — a non-structural metadata write that doesn't affect sort order doesn't bump treeVersion, so the popover doesn't even recompute. The counter is a hand-tuned "did anything I care about change?" signal.

Decision Guide 决策指南

Does your selector return a freshly-computed object/array?
  ├─ yes → don't subscribe it directly (unstable ref → over-renders)
  │       subscribe a version counter, read via getState() in useMemo
  └─ no (primitive/stable ref) → subscribe it directly, it's fine

Alternative: pass a custom equality fn (e.g. shallow) to the selector —
use when you can't add a counter to the store you don't own.

Quiz 测验 — scenario-based 情景题

每题给一个具体场景，选出你会怎么做。重点不是「概念是什么」，而是「在这个场景里你选哪个、为什么」。

Q1. You're adding a "favorite color theme" toggle to the sidebar. It's a pure UI preference. How should you store it? / 你要加一个纯 UI 的主题色偏好开关，怎么存？

Key it by userId like the pinned ids. 像置顶 id 一样按用户分键。 Device-global key — it's a UI toggle, not account data. 设备级全局 key——是 UI 偏好不是账号数据。 Store it server-side via an API. 通过 API 存服务端。 Use a module-level variable. 用模块级变量。
A theme toggle is a property of the device/browser, not the account — so a device-global key (like pinnedProjects' isExpanded) is correct. Only data that references server entities needs per-user scoping. 主题偏好是设备属性而非账号属性，设备级 key 正确。只有引用服务端实体的数据才需要按用户分键。

Q2. A Radix popover re-fetches a list on every open. The data lives in a Zustand store. Best fix? / Radix 弹窗每次打开都重拉列表，数据在 Zustand store 里，最佳修复？

Add a module-level cache variable. 加模块级缓存变量。 Keep the popover content mounted (forceMount). 让弹窗内容常驻挂载。 Reuse the store + a session flag to skip the per-open refetch. 复用 store + 会话标志挡掉每次打开的请求。 Switch to React Query. 换成 React Query。
If the data is already in a store, the store survives remounts for free — you only need a session flag (hasFetchedTurboWorkflowsOnce) to skip the redundant per-open fetch. A module cache would be redundant; forceMount changes UX semantics. 数据已在 store 里，store 本就能跨重挂载存活，只需一个会话标志挡掉冗余请求。

Q3. Two components share an ordered "pinned + rest" split. A third only needs "is X pinned?". What do you extract? / 两个组件共享「置顶+余项」有序拆分，第三个只需「X 是否置顶」。抽什么？

Extract splitByPinned for the two; leave the third's includes() inline. 给前两个抽 splitByPinned；第三个保留 includes()。 Extract one mega-helper covering all three. 抽一个覆盖三者的大 helper。 Don't extract — duplication is fine everywhere. 都不抽，到处重复也行。 Extract three separate tiny helpers. 抽三个独立小 helper。
Extract where the same business RULE is shared (the ordered split). The boolean membership test is a genuinely different shape — folding it into the shared selector would be a worse abstraction. Extract by shared rule, not shared keyword. 按共享的业务规则抽取（有序拆分）；布尔成员判断形状不同，强行合并是更差的抽象。

Q4. A Zustand selector s.getRootFolders() returns a new array each call, causing over-renders. Cleanest fix in a store you own? / 选择器每次返回新数组导致过度渲染，你拥有该 store，最干净的修复？

Wrap the component in React.memo. 用 React.memo 包组件。 Call getRootFolders() in render without useMemo. 渲染里直接调用不用 memo。 Deep-clone and compare manually. 手动深拷贝比较。 Subscribe a treeVersion counter, read via getState() in a useMemo keyed on it. 订阅 treeVersion 计数器，在以它为 key 的 useMemo 里用 getState() 读取。
Subscribe a stable integer that bumps only on real change, then read the derived data imperatively inside useMemo. React.memo doesn't help (the unstable ref is the prop); calling in render still re-runs every render. If you didn't own the store, a custom equality fn would be the fallback. 订阅只在真正变化时自增的整数，再在 memo 内命令式读取派生数据。

Q5. Your module cache survives remounts, but a workflow gets renamed elsewhere and the cached list shows the old name. What's missing? / 模块缓存能存活重挂载，但别处重命名后缓存仍显示旧名。缺了什么？

A longer cache TTL. 更长的缓存 TTL。 Event-driven in-place invalidation (patch the cache on a store event). 事件驱动的原地失效（监听 store 事件打补丁）。 Disable the cache entirely. 干脆禁用缓存。 Poll the API every 5 seconds. 每 5 秒轮询一次。
"Survive remount" and "stay fresh" are separate problems. The cache survives fine — it just needs freshness via store-event subscription (MA_WORKFLOW_CHANGED patches the cache in place), not polling or disabling. 「存活」和「新鲜」是两个问题；缓存没问题，缺的是事件驱动的新鲜度维护。

Q6. You ship a per-user storage key. Existing users had their data under the old global key. What must you NOT forget? / 你上线了按用户分键的存储，老用户数据在旧全局 key 下。绝不能忘的是？

One-time migration of the legacy key on first sync to a real user. 首次同步到真实用户时一次性迁移旧 key。 Nothing — old data can just be lost. 没什么，旧数据丢了就丢了。 Force every user to re-login. 强制所有用户重新登录。 Keep writing to both keys forever. 永远同时写两个 key。
Without migration, every existing user silently loses their pins on deploy. The pattern: on first sync to a real user, if the new key is empty, adopt the legacy global value then delete it. Writing to both forever defeats the scoping fix. 不迁移则老用户上线即丢数据；正确做法是首次同步时迁移旧值再删除旧 key。

项目规则：不要仅为满足行数指标去拆文件/抽 helper。抽取的理由必须是「2+ 处共享同一业务规则」。
Line count is never a reason to extract (project CLAUDE.md rule).
Unstable selector reference
返回新数组/对象的 selector 每次渲染产生新引用，Zustand 用 Object.is 比较永远不相等 → 过度渲染。
A selector returning a fresh array is never === the previous one → over-renders.
Version-counter signal
订阅一个只在真正变化时自增的整数（treeVersion），在以它为 key 的 useMemo 内用 getState() 命令式读取派生数据。store 还能选择性自增（无关变化不 bump）。
Subscribe a counter, read derived data via getState() in a keyed useMemo.
Custom equality fn (fallback)
当你无法给不属于自己的 store 加版本号时，给 selector 传一个自定义 equality（如 shallow）来抑制无效重渲染——是 version-counter 的退路方案。
Pass a shallow/custom equality fn when you can't add a counter to a store you don't own.
