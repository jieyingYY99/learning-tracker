UIBlock Interactive Cards Pattern
UIBlock 交互卡片模式
Frontend 前端

Parse structured UI directives (ask_user, suggest_connectors) from SSE streams and render them as interactive card components. User actions call the ask_user/respond API to return results to the backend agent, enabling AI-driven multi-step interaction flows.

在 AI Chat 的 SSE 流中解析结构化 UI 指令（ask_user、suggest_connectors），渲染为交互卡片组件（AskUserCard、SuggestConnectorCard），用户操作后调用 ask_user/respond API 回传结果给后端 agent，实现 AI 驱动的多步交互流程。

Key Points 要点

  • Stream parsing 流解析: streamProcessor detects UIBlock types in SSE chunks and routes to appropriate card components
    streamProcessor 在 SSE 数据块中检测 UIBlock 类型并路由到对应的卡片组件
  • Response pattern 响应模式: Each card submits via response_key to ask_user/respond API endpoint
    每个卡片通过 response_key 向 ask_user/respond API 端点提交
  • Component separation 组件分离: AskUserCard (selection + custom input + skip) and SuggestConnectorCard are independent, reusable UIBlock renderers
    AskUserCard（选择 + 自定义输入 + 跳过）和 SuggestConnectorCard 是独立、可复用的 UIBlock 渲染器
  • Type-driven rendering 类型驱动渲染: Chat types.ts defines UIBlock interfaces that drive which component renders
    Chat types.ts 定义了 UIBlock 接口来驱动组件渲染
// streamProcessor.ts — parsing UIBlock from SSE
if (block.type === 'ask_user') {
  return <AskUserCard options={block.options} responseKey={block.response_key} />
}

// AskUserCard — submitting response
const handleSubmit = async (value: string) => {
  await api.post('/ask_user/respond', {
    response_key: responseKey,
    value
  });
};

Pipedream Connect OAuth Integration
第三方 OAuth 集成（Pipedream Connect）
Backend 后端

Implement third-party OAuth via Pipedream Connect SDK: backend API route generates connect_token bound to user ID (preventing cross-user leaks), frontend PipedreamProvider wraps createFrontendClient, ConnectModal triggers OAuth popup, with full account lifecycle management.

通过 Pipedream Connect SDK 实现第三方服务 OAuth 授权：后端 API route 生成 connect_token（绑定用户 ID 防止跨用户泄漏）、前端 PipedreamProvider 包裹 createFrontendClient、ConnectModal 触发 OAuth 弹窗、完整的账户生命周期管理。

Key Points 要点

  • Security 安全性: connect_token must be bound to authenticated user ID — cookie key mismatch caused cross-user account leaks
    connect_token 必须绑定已认证的用户 ID — cookie key 不匹配导致了跨用户账户泄漏
  • Token flow 令牌流程: Server route reads real user ID from session cookie → generates short-lived connect_token → frontend SDK uses it to open OAuth popup
    服务端路由从 session cookie 读取真实用户 ID → 生成短时效 connect_token → 前端 SDK 用它打开 OAuth 弹窗
  • Provider pattern: PipedreamProvider at app root creates singleton frontend client for all connector components
    PipedreamProvider 在应用根部创建单例前端客户端供所有连接器组件使用
  • Account lifecycle 账户生命周期: Connect → list accounts → disconnect → reconnect, with auto-close modal on success
    连接 → 列出账户 → 断开 → 重新连接，成功后自动关闭弹窗
// app/api/connectors/token/route.ts
export async function POST(req: Request) {
  const userId = getUserIdFromCookie(req); // Must be real user ID!
  const token = await pipedream.createConnectToken({
    external_user_id: userId,
  });
  return Response.json({ token: token.token });
}

Customize View Tab Architecture
可定制化视图架构
Frontend 前端

Build a dual-tab management interface (Skills + Connectors): SkillsTab supports Official/Custom filtering with detail dialog featuring a file tree browser; ConnectorsTab integrates Pipedream curated apps with search, skeleton loading, and lazy-mount tabs for optimized initial render.

构建 Skills + Connectors 双标签页管理界面：SkillsTab 支持 Official/Custom 分类过滤、技能详情弹窗含文件树浏览器；ConnectorsTab 集成 Pipedream curated apps 列表、搜索过滤、skeleton loading、lazy-mount tabs 优化首次渲染。

Key Points 要点

  • Lazy-mount tabs 懒挂载标签页: Only mount tab content when first selected, avoiding heavy Pipedream SDK init on page load
    仅在首次选中标签页时挂载内容，避免页面加载时初始化重量级 Pipedream SDK
  • Skeleton loading: Show content placeholders while fetching skills/connectors data for perceived performance
    在获取数据时显示内容占位符以提升感知性能
  • File tree browser 文件树浏览器: SkillDetailDialog renders skill files as collapsible tree with expand/collapse directories
    SkillDetailDialog 将技能文件渲染为可折叠的树形目录结构
  • Radix primitives: Replaced raw buttons/dropdowns with Button component and Radix Dialog/Dropdown for accessibility
    用 Button 组件和 Radix Dialog/Dropdown 替换原生按钮/下拉菜单以提升无障碍性

Download/Export System Refactoring
下载导出系统重构
Frontend 前端

Consolidate scattered download logic into FolderWidgets/Download module: split handlers by file type (fromMarkdown, fromHtml, fromPptx), unify PDF export pipeline (Tiptap → HTML → PDF with TOC, ref-links, heading processing), and abstract URL-based download options.

将分散在各组件中的下载逻辑统一到 FolderWidgets/Download 模块：按文件类型拆分 handler（fromMarkdown、fromHtml、fromPptx）、统一 PDF 导出管线（Tiptap → HTML → PDF，含 TOC、ref-link、heading 处理）、URL-based 下载选项抽象。

Key Points 要点

  • Pipeline architecture 管线架构: Markdown → Tiptap DOM → HTML string → PDF pages, with each step independently testable
    Markdown → Tiptap DOM → HTML 字符串 → PDF 页面，每步可独立测试
  • Heading with ID: HeadingWithId extension generates stable IDs for TOC navigation in exported PDFs
    HeadingWithId 扩展生成稳定 ID 用于导出 PDF 中的目录导航
  • Ref-link resolution: RefLinkMark resolves internal document references to proper URLs during export
    RefLinkMark 在导出时将内部文档引用解析为正确的 URL
  • Escape handling 转义处理: Special characters in headings must be escaped properly for PDF rendering
    标题中的特殊字符必须正确转义以用于 PDF 渲染

积分争议系统
General 通用

Build a complete credit dispute workflow: DisputeDialog for submitting dispute reasons, API layer supporting dispute creation and status queries (pending/approved/rejected), Usage panel integration with status icons, and GA4 event tracking for user behavior analytics.

    三种状态（pending/approved/rejected），在 Usage 面板中用可视图标展示

    GA4 自定义事件追踪争议创建、解决和用户流程，用于广告归因

1. In the UIBlock pattern, how does the frontend communicate the user's selection back to the backend agent?
在 UIBlock 模式中，前端如何将用户的选择传回后端 agent？
A. WebSocket message with selection data
WebSocket 消息携带选择数据
B. POST to ask_user/respond API with response_key and value
POST 到 ask_user/respond API 携带 response_key 和 value
C. Append selection to the next chat message
将选择追加到下一条聊天消息
D. Store in Zustand and let SSE pick it up
存入 Zustand 让 SSE 获取
The UIBlock cards use a dedicated REST endpoint (ask_user/respond) with a response_key to match the response to the original question. This decouples the interactive flow from the chat stream.
UIBlock 卡片使用专用 REST 端点（ask_user/respond）配合 response_key 将响应匹配到原始问题。这将交互流程与聊天流解耦。
2. What critical security issue was discovered in the Pipedream Connect integration?
在 Pipedream Connect 集成中发现了什么关键安全问题？
A. OAuth tokens were stored in localStorage
OAuth 令牌存储在 localStorage 中
B. connect_token had no expiry
connect_token 没有过期时间
C. Wrong cookie key caused cross-user connector account leaks
错误的 cookie key 导致跨用户连接器账户泄漏
D. OAuth callback URL was not validated
OAuth 回调 URL 未经验证
Using the wrong cookie key to extract user ID meant connect_tokens could be generated for the wrong user, allowing one user to see another user's connected accounts. Fixed by using the correct session cookie key.
使用错误的 cookie key 提取用户 ID 意味着 connect_token 可能为错误的用户生成，导致一个用户能看到另一个用户的已连接账户。通过使用正确的 session cookie key 修复。
3. Why do the Customize View tabs use "lazy-mount" instead of rendering all tabs at once?
为什么 Customize View 的标签页使用"懒挂载"而不是一次渲染所有标签页？
A. To reduce bundle size with code splitting
通过代码分割减少包体积
B. To avoid initializing heavy Pipedream SDK and fetching data for hidden tabs
避免为隐藏的标签页初始化重量级 Pipedream SDK 和获取数据
C. Because Radix Tabs requires it
因为 Radix Tabs 要求这样做
D. To support server-side rendering
为了支持服务端渲染
The Connectors tab initializes the Pipedream frontend client and fetches curated apps data. Lazy-mounting ensures this heavy work only happens when the user actually clicks the Connectors tab, improving initial page load.
Connectors 标签页会初始化 Pipedream 前端客户端并获取 curated apps 数据。懒挂载确保这些重量级操作仅在用户实际点击 Connectors 标签页时才发生，提升了首次页面加载速度。
4. In the download/export refactoring, what is the PDF export pipeline order?
在下载导出重构中，PDF 导出管线的顺序是什么？
A. Markdown → PDF → TOC injection
Markdown → PDF → TOC 注入
B. HTML → Canvas → PDF pages
HTML → Canvas → PDF 页面
C. Tiptap DOM → HTML string → PDF pages (with TOC, ref-links, heading processing)
Tiptap DOM → HTML 字符串 → PDF 页面（含 TOC、ref-link、heading 处理）
D. Markdown → AST → LaTeX → PDF
Markdown → AST → LaTeX → PDF
The pipeline converts Tiptap's ProseMirror document to HTML, then processes headings (with IDs for TOC), resolves ref-links, escapes special characters, and renders to PDF pages. Each step is modular and testable.
管线将 Tiptap 的 ProseMirror 文档转换为 HTML，然后处理标题（含 ID 用于 TOC）、解析 ref-link、转义特殊字符，最后渲染为 PDF 页面。每一步都是模块化和可测试的。
5. Why was the dispute API path migrated from /credits/dispute to /dispute?
为什么 dispute API 路径从 /credits/dispute 迁移到 /dispute？
A. To fix a routing conflict with the credits endpoint
修复与 credits 端点的路由冲突
B. To keep API paths clean and resource-oriented — disputes are their own resource
保持 API 路径干净且面向资源 — disputes 是独立资源
C. The backend team required a versioned API structure
后端团队要求版本化的 API 结构
D. To enable caching at the CDN level
为了在 CDN 层启用缓存
Disputes, while related to credits, are their own distinct resource with independent CRUD operations and status lifecycle. Nesting them under /credits implied they were a sub-operation rather than a first-class entity.
Disputes 虽然与 credits 相关，但它们是独立的资源，有自己的 CRUD 操作和状态生命周期。将它们嵌套在 /credits 下暗示它们是子操作而非一等实体。
