Batch Operations Design Pattern
批量操作设计模式
FrontendNEW

Design-doc-driven batch delete for Trash: separate endpoints for "empty all" vs "delete selected", idempotent API responses with cleared_file_ids, store-level selection via Set<number>, and local Radix dialogs for confirmation.

设计文档驱动的 Trash 批量删除：区分"清空全部"和"删除选中"的独立端点、带 cleared_file_ids 的幂等 API 响应、store 层 Set<number> 选择管理、本地 Radix Dialog 确认。

Key Points 要点

  • Separate endpoints 独立端点: POST /trash-bin/empty (no body) vs POST /trash-bin/delete (file_ids array). Keeps semantics clean, avoids parameter explosion.
    POST /trash-bin/empty（无请求体）vs POST /trash-bin/delete（file_ids 数组）。保持语义清晰，避免参数膨胀。
  • Idempotent response 幂等响应: Server returns cleared_file_ids — frontend doesn't guess which rows transitioned, just filters by returned IDs.
    服务端返回 cleared_file_ids — 前端不用猜测哪些行被操作了，直接按返回的 ID 过滤。
  • Store-level selection Store 层选择: selectedIds: Set<number> lives in trash store, not component state. UI is purely presentational.
    selectedIds: Set<number> 存在 trash store 中，非组件状态。UI 纯展示。
  • Mutation flags 操作标志: isDeleting, isEmptying, isRestoring — all three disable toolbar buttons via isBusy computed value.
    isDeleting、isEmptying、isRestoring — 三个标志通过 isBusy 计算值禁用工具栏按钮。

Race Condition Prevention Patterns
竞态条件防护模式集
FrontendNEW

Five distinct patterns for preventing race conditions in React + async workflows: version counter (invalidate stale loads), dual guard (state + ref), boolean flag (concurrent prevention), AbortController (stream cancellation), and timer suppression (event debounce).

React + 异步工作流中防止竞态的五种模式：版本计数器（作废过时加载）、双重守卫（state + ref）、布尔标志（并发防护）、AbortController（流取消）、定时器抑制（事件防抖）。

Key Points 要点

  • Bug 1 — Missing await: newConversationHandler() wasn't awaited, so message sent before conversation existed on server. Fix: make it async + await.
    Bug 1 — 缺少 await：newConversationHandler() 未被 await，导致消息在会话创建前就发送。修复：改为 async + await。
  • Bug 2 — Stale history overwrite: loadConversationMessages races with createConversation. Fix: loadVersionRef counter, increment on create, check on load completion.
    Bug 2 — 过时历史覆盖：loadConversationMessages 与 createConversation 竞争。修复：loadVersionRef 计数器。
  • Bug 3 — Duplicate sends: Error catch cleared ref but not state, causing re-fire. Fix: clear both sendToAgent(null) + processedPayloadRef = null.
    Bug 3 — 重复发送：错误捕获清除了 ref 但未清除 state，导致重新触发。修复：同时清除两者。
  • Bug 4 — Drag-select lost: 200ms suppress timer expired before commitAndReset. Fix: re-arm with 300ms on commit.
    Bug 4 — 拖选丢失：200ms 抑制定时器在 commitAndReset 前过期。修复：在 commit 时重新设置 300ms。

Which race condition pattern? 选哪种竞态防护？
├─ Need to cancel in-flight network request? 需要取消进行中的网络请求？
→ AbortController (passes signal to fetch/EventSource)
├─ New operation should invalidate old result? 新操作应作废旧结果？
→ Version counter (loadVersionRef++, check at completion)
├─ Same effect re-fires with identical data? 相同 effect 用相同数据重新触发？
→ Dual guard (state null check + ref dedup by serialized key)
├─ Prevent button double-click / concurrent calls? 防止按钮重复点击/并发调用？
→ Boolean flag (isProcessingRef)
└─ Suppress post-transition events? 抑制过渡后事件？
→ Timer suppression (setTimeout + re-arm on state change)

const loadVersionRef = useRef(0);
const load = async () => {
  const v = loadVersionRef.current;
  const data = await fetchData();
  if (v !== loadVersionRef.current) return; // Stale!
  setData(data);
};
const create = () => { loadVersionRef.current++; /* ... */ };

// Pattern 2: Dual Guard
const processedRef = useRef<string | null>(null);
useEffect(() => {
  if (!payload) return;                           // Guard 1
  const key = JSON.stringify(payload);
  if (key === processedRef.current) return;       // Guard 2
  processedRef.current = key;
  try { await process(payload); }
  finally { setPayload(null); processedRef.current = null; }
}, [payload]);

Tiptap InputRule for Slash Commands
Tiptap InputRule 斜杠命令内联节点
FrontendNEW

Custom Tiptap Node triggered by InputRule regex: when user types /skillName followed by space, the text is replaced with an atomic, non-editable pill node. Fixed regex to account for the trailing space already being inserted before InputRule fires.

通过 InputRule 正则触发的 Tiptap 自定义节点：用户输入 /skillName 加空格后，文本被替换为不可编辑的原子标签节点。修复了正则以处理 InputRule 触发前空格已被插入的问题。

Key Points 要点

  • Regex timing 正则时序: Tiptap InputRules fire AFTER the character is inserted. So /skill<space> needs regex ending with \s$, not just matching /skill$.
    Tiptap InputRules 在字符插入后触发。所以 /skill<空格> 需要正则以 \s$ 结尾。
  • Range calculation 范围计算: Use indexOf() to find exact position of /skillName in matched text, not endsWith() which breaks with leading whitespace.
    用 indexOf() 找到 /skillName 的精确位置，而非 endsWith()（在有前导空格时会出错）。
  • Skill validation 技能验证: Before creating the node, check skillNameSet from store to ensure it's a real skill name.
    创建节点前，检查 store 中的 skillNameSet 确保是真实的技能名。
  • Atom node 原子节点: atom: true makes the node non-editable, non-splittable — it behaves like a single character that can only be deleted whole.
    atom: true 使节点不可编辑、不可分割 — 行为像单个字符，只能整体删除。

// InputRule pattern: fires AFTER character insertion
const SLASH_TRIGGER_REGEX = /(?:^|\s)\/([^\s/]+)\s$/;
//                                                 ^^ trailing space already inserted!

new InputRule({
  find: SLASH_TRIGGER_REGEX,
  handler: ({ chain, match, range }) => {
    const skillName = match[1];
    if (!skillNameSet.has(skillName)) return;  // Validate against store

    // Use indexOf for precise range (handles leading whitespace)
    const slashIndex = match[0].indexOf(`/${skillName}`);
    const from = range.from + (slashIndex >= 0 ? slashIndex : 0);

    chain()
      .deleteRange({ from, to: range.to })
      .insertContentAt(from, [{ type: 'inlineSlashNode', attrs: { slashName: skillName } }])
      .run();
  },
});

Scenario-Based Quiz 场景测验

Decision-making questions — "Which pattern would you choose?" not "What is X?"
决策类题目 — 问"你会选哪种方案？"而非"X是什么？"

1. You're building a "permanently delete selected bookmarks" feature. The operation is irreversible. Which pattern from the codebase should you follow?
你正在构建"永久删除选中书签"功能。操作不可逆。应该遵循代码库中的哪种模式？
Context: Users select items from a list, click "Delete", confirm, items are gone forever. Similar to Trash batch delete.
A) Finder delete pattern: optimistic update + UndoEntry for rollback
B) Finder move pattern: per-item conflict dialog with overwrite/skip/rename
C) Trash delete pattern: explicit confirmation dialog + idempotent API returning cleared_file_ids
D) No confirmation needed — just delete on click
For irreversible operations, you need: (1) explicit confirmation (Radix Dialog, not just optimistic update), (2) idempotent API response so the frontend knows exactly what was deleted, (3) no undo support needed since there's nothing to undo. The Trash pattern was designed exactly for this.
对于不可逆操作：(1) 需要明确确认，(2) 幂等 API 响应让前端知道确切删除了什么，(3) 无需撤销支持。Trash 模式正是为此设计的。
2. You have an async function that fetches conversation history. A user switches conversations rapidly. How do you prevent the first conversation's late-arriving data from overwriting the second?
你有一个获取对话历史的异步函数。用户快速切换对话。如何防止第一个对话延迟到达的数据覆盖第二个？
A) Boolean flag (isLoadingRef) to block concurrent loads
B) Version counter — increment on each switch, check version at completion, discard if stale
C) AbortController to cancel the first fetch
D) Debounce the conversation switch by 500ms
Version counter is the best fit: increment loadVersionRef on each switch, capture the version before await, check after — if it doesn't match, the result is stale and should be discarded. AbortController (C) could work too but doesn't guarantee the response won't arrive; boolean flag (A) would block legitimate loads; debounce (D) adds latency.
版本计数器最合适：每次切换时递增 loadVersionRef，await 前捕获版本号，完成后检查 — 如果不匹配则结果过时应丢弃。
3. A useEffect re-fires with the same sendToAgentPayload due to React batching. You need to prevent processing the same payload twice. What's the most robust approach?
由于 React 批处理，useEffect 用相同的 sendToAgentPayload 重新触发。你需要防止同一 payload 处理两次。
A) Dual guard: state null check + ref-based dedup via JSON.stringify key, clear BOTH in try/catch
B) Just check if payload is null before processing
C) Use useRef to track if effect already ran
D) Add the payload to useEffect dependency array
The dual guard pattern is needed because: (1) null check alone doesn't prevent duplicate processing of the same non-null payload, (2) a simple "already ran" ref doesn't distinguish between different payloads, (3) the JSON.stringify key ensures each unique payload is processed exactly once, (4) clearing BOTH state AND ref in catch prevents the error-retry-duplicate bug that was fixed in commit 454210ba.
双重守卫：(1) 仅 null 检查不能防止同一非空 payload 的重复处理，(2) JSON.stringify key 确保每个唯一 payload 只处理一次，(3) catch 中同时清除 state 和 ref 防止错误重试重复。
4. You need to create a Tiptap inline element that auto-triggers when the user types @mention followed by a space. Which approach matches the InlineSlashNode pattern?
你需要创建一个 Tiptap 内联元素，在用户输入 @mention 加空格后自动触发。哪种方案匹配 InlineSlashNode 模式？
A) Listen for keydown events and manually insert the node
B) Use a Tiptap Suggestion plugin (like prosemirror-suggest)
C) InputRule with regex /@([^\s@]+)\s$/, validate against user list, then deleteRange + insertContentAt
D) Use ReactNodeViewRenderer with an onChange handler
The InlineSlashNode pattern: (1) InputRule with regex that includes trailing \s$ (because character is already inserted when rule fires), (2) validate the captured name against a known set, (3) use indexOf() for precise range calculation, (4) chain deleteRange + insertContentAt to replace text with atom node. This pattern works for any "type X + trigger char → convert to node" interaction.
InlineSlashNode 模式：(1) 包含尾随 \s$ 的 InputRule 正则，(2) 验证捕获的名称，(3) indexOf() 精确计算范围，(4) chain deleteRange + insertContentAt。
5. After a drag-and-drop operation, the user's pointer-up event triggers a click that clears the selection. Your 200ms suppress timer already expired during the drag. What's the fix?
拖放操作后，用户的 pointer-up 事件触发了一个 click，清除了选区。你的 200ms 抑制定时器在拖拽期间已过期。怎么修？
A) Increase the suppress timer to 2000ms to cover all drag durations
B) Re-arm the suppress timer in commitAndReset with a ≥ 300ms duration (longer than the click delay)
C) Use e.stopPropagation() on the drag end event
D) Check if items are selected in the click handler and skip if yes
The fix is to re-arm the suppress flag when the drag commits (commitAndReset), not just when it starts. The START timer (200ms) prevents clicks during very short drags. The END timer (300ms) prevents the post-drag click that arrives after commitAndReset. Both timers are needed. Making it too long (A) would suppress legitimate clicks.
修复方法：在拖拽提交时（commitAndReset）重新设置抑制标志，而非仅在开始时。START 定时器（200ms）防止极短拖拽时的点击。END 定时器（300ms）防止 commitAndReset 后的拖拽后点击。
6. You're designing a "bulk restore from trash" feature. Restore can partially fail (some files' target folders were deleted). Which API response pattern should you use?
你在设计"从回收站批量恢复"功能。恢复可能部分失败（某些文件的目标文件夹已删除）。应该用哪种 API 响应模式？
A) Return both successfulIds AND failedIds arrays, like the Finder move pattern — frontend removes successes, shows errors for failures
B) Return only cleared_file_ids like trash delete — simpler, let frontend figure out failures
C) All-or-nothing: if any fail, roll back all and return error
D) Return just a boolean success/failure flag
When partial failure is expected (like restoring to potentially-deleted folders), you need the Finder move pattern: return both successfulIds and failedIds so the frontend can handle each case appropriately — remove restored items from trash list, show error messages for failures, and potentially offer "create missing folder?" for failed items. The trash delete pattern (B) works when partial failure is unexpected.
当部分失败可预期时，需要 Finder move 模式：返回 successfulIds 和 failedIds，前端分别处理 — 从回收站移除已恢复项，为失败项显示错误消息。
7. You need to prevent a Pipedream connect button from being clicked multiple times while OAuth popup is open. Which pattern?
你需要防止 Pipedream 连接按钮在 OAuth 弹窗打开期间被多次点击。用哪种模式？
A) Version counter pattern
B) Dual guard with JSON.stringify dedup
C) Timer suppression
D) Boolean flag: connectingRef.current = true in try, false in finally
Boolean flag is simplest and most appropriate for preventing concurrent execution of the same async function. Set flag at start, clear in finally block. No need for version counters (not superseding), dedup (not same-payload issue), or timers (not event suppression). This is exactly what ConnectorsTab uses.
布尔标志最简单也最适合防止同一异步函数的并发执行。开始时设置，finally 中清除。
8. Your Tiptap InputRule regex /\/([^\s]+)$/ doesn't trigger when user types "/skill " (with trailing space). Why?
你的 Tiptap InputRule 正则 /\/([^\s]+)$/ 在用户输入 "/skill "（带尾随空格）时不触发。为什么？
A) The regex is missing the ^ anchor
B) Tiptap InputRules only fire on Enter, not on Space
C) InputRules fire AFTER character insertion — the space is already in the text, so $ doesn't match the position before the space
D) The [^\s]+ is too greedy
Tiptap InputRules run after the character is inserted into the document. When the user presses space, the text becomes "/skill " (with space) before the rule evaluates. The regex [^\s]+$ expects the match to end at the cursor with no whitespace, but there's now a space there. Fix: change to /\/([^\s]+)\s$/ to include the trailing space in the match.
Tiptap InputRules 在字符插入后运行。按空格时文本变为 "/skill "（含空格），然后规则才评估。[^\s]+$ 期望匹配在光标处结束且无空格，但现在有空格了。修复：改为包含尾随空格。
