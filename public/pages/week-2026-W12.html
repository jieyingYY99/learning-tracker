Virtual Scroll with Tree Flattening虚拟滚动与树形扁平化

Migrating from recursive DOM rendering to flat array + react-window从递归 DOM 渲染迁移到扁平数组 + react-window

The Problem问题

A deeply nested folder tree with hundreds of nodes creates a massive DOM. Every folder expansion re-renders the entire tree. Users with large workspaces experienced visible jank and slow interactions.深层嵌套的文件夹树包含数百个节点，会创建庞大的 DOM。每次展开文件夹都会重新渲染整棵树。大型工作空间的用户会经历明显的卡顿和缓慢的交互。

Before: Recursive Rendering之前：递归渲染

Each folder component recursively renders its children. 500 nodes = 500 mounted DOM elements. Opening a folder triggers a cascade of re-renders through the entire tree.每个文件夹组件递归渲染其子节点。500 个节点 = 500 个挂载的 DOM 元素。打开文件夹会触发整棵树的连锁重新渲染。

After: Virtual Scroll之后：虚拟滚动

The tree is flattened into a single array. react-window only mounts the ~20 visible rows. Opening a folder inserts items into the flat array -- no recursive DOM traversal.将树扁平化为单个数组。react-window 只挂载约 20 个可见行。打开文件夹只是在扁平数组中插入项目——无需递归 DOM 遍历。

Flattening the Tree扁平化树结构

The useFlattenedTree hook walks the tree recursively and produces a flat FlatTreeItem[] array. Each item carries its depth, parentId, and isExpanded state so the list can render indentation and toggle icons without knowing the tree structure.useFlattenedTree hook 递归遍历树并生成扁平的 FlatTreeItem[] 数组。每个项目携带 depth、parentId 和 isExpanded 状态，使列表无需了解树结构即可渲染缩进和折叠图标。

interface FlatTreeItem {
  id: string;
  name: string;
  type: 'folder' | 'file';
  depth: number;
  parentId: string | null;
  isExpanded: boolean;
  hasChildren: boolean;
  isLoading: boolean;   // skeleton while fetching children
}

function useFlattenedTree(root: TreeNode, expanded: Set<string>): FlatTreeItem[] {
  return useMemo(() => {
    const result: FlatTreeItem[] = [];

    function walk(node: TreeNode, depth: number, parentId: string | null) {
      const isExp = expanded.has(node.id);
      result.push({
        id: node.id,
        name: node.name,
        type: node.type,
        depth,
        parentId,
        isExpanded: isExp,
        hasChildren: node.children?.length > 0,
        isLoading: node.isLoading ?? false,
      });

      if (isExp && node.children) {
        for (const child of node.children) {
          walk(child, depth + 1, node.id);
        }
      }
    }

    walk(root, 0, null);
    return result;
  }, [root, expanded]);
}

O(1) Lookup with useIndexTree使用 useIndexTree 实现 O(1) 查找

Once flattened, finding a node by ID in the flat array is O(n). The useIndexTree hook builds a Map<string, number> mapping each ID to its index in the flat array. This gives O(1) lookup for scroll-to-item, selection, and keyboard navigation.扁平化后，在数组中按 ID 查找节点是 O(n)。useIndexTree hook 构建了一个 Map<string, number>，将每个 ID 映射到其在数组中的索引，实现 O(1) 查找，用于滚动定位、选择和键盘导航。

function useIndexTree(items: FlatTreeItem[]): Map<string, number> {
  return useMemo(() => {
    const map = new Map<string, number>();
    items.forEach((item, index) => map.set(item.id, index));
    return map;
  }, [items]);
}

Loading Skeletons加载骨架屏

When a folder is expanded but its children have not been fetched yet, the flat array includes a single placeholder item with isLoading: true. The row renderer shows an animated skeleton at the correct indentation depth. Once the fetch completes, the placeholder is replaced by real items and react-window seamlessly updates the list height.当文件夹已展开但子节点尚未加载时，扁平数组中会包含一个 isLoading: true 的占位项。行渲染器在正确的缩进深度显示动画骨架屏。加载完成后，占位项被真实数据替换，react-window 无缝更新列表高度。

Key Insight 核心洞察

Virtual scroll does not just improve performance by reducing initial render cost. It fundamentally changes the rendering model: the tree structure lives in data, not in the DOM. This makes operations like search-and-scroll, keyboard navigation, and drag reordering much simpler because they all work on a single flat array.虚拟滚动不仅通过减少初始渲染成本来提升性能，更从根本上改变了渲染模型：树结构存在于数据中，而非 DOM 中。这使得搜索定位、键盘导航和拖拽排序等操作变得更简单，因为它们都在单个扁平数组上操作。

Folder Highlight on Drag-Over拖拽悬停时文件夹高亮

When a dragged item hovers over a folder row, the folder needs a distinct visual indicator (background color change, border highlight). This is achieved by tracking the overId from onDragOver and checking whether the hovered item is a folder. The highlight state is passed as a prop to the row renderer.当拖拽项悬停在文件夹行上时，需要明显的视觉指示（背景色变化、边框高亮）。通过跟踪 onDragOver 的 overId 并检查目标是否为文件夹来实现。高亮状态作为 prop 传递给行渲染器。

const handleDragOver = (event: DragOverEvent) => {
  const { over } = event;
  if (over && isFolder(over.id)) {
    setHighlightedFolderId(over.id as string);
  } else {
    setHighlightedFolderId(null);
  }
};

Auto-Expand Folders on Hover悬停自动展开文件夹

A key UX pattern: when a user drags an item over a collapsed folder for 500ms, the folder automatically expands. This uses a setTimeout that is cleared whenever the drag leaves the folder or a drop occurs.关键 UX 模式：用户将拖拽项悬停在折叠的文件夹上 500ms 后，文件夹自动展开。使用 setTimeout 实现，当拖拽离开文件夹或放置时清除计时器。

const hoverTimerRef = useRef<NodeJS.Timeout | null>(null);

const handleDragOver = (event: DragOverEvent) => {
  const { over } = event;

  // Clear any pending timer
  if (hoverTimerRef.current) {
    clearTimeout(hoverTimerRef.current);
    hoverTimerRef.current = null;
  }

  if (over && isFolder(over.id) && !isExpanded(over.id)) {
    hoverTimerRef.current = setTimeout(() => {
      expandFolder(over.id as string);
    }, 500);
  }
};

// Cleanup on drop or drag end
const handleDragEnd = () => {
  if (hoverTimerRef.current) clearTimeout(hoverTimerRef.current);
  setHighlightedFolderId(null);
};

One of the trickiest aspects of a tree-based drag-and-drop is distinguishing between "drop INTO a folder" and "drop BETWEEN two items" (reorder). The approach: divide each row into three vertical zones.基于树的拖放中最棘手的一点是区分"放入文件夹"和"在两个项目之间插入"（重新排序）。方法是将每行划分为三个垂直区域。

Undo Support撤销支持

Before executing a move, snapshot the current tree state. If the move fails or the user presses Ctrl+Z, restore the snapshot. The undo stack stores a closure that calls the inverse API operation and restores the UI state simultaneously.执行移动前，快照当前树状态。如果移动失败或用户按 Ctrl+Z，恢复快照。撤销栈存储一个闭包，同时调用逆向 API 操作并恢复 UI 状态。

Key Insight 核心洞察

Drop precision requires careful calculation of the pointer's Y position relative to the drop target row. Using event.activatorEvent and the droppable's rect, you can compute a ratio (0-1) within the row and map it to the three zones. Getting this wrong makes the UI feel unpredictable.放置精度需要精确计算指针相对于目标行的 Y 位置。使用 event.activatorEvent 和 droppable 的 rect，可以计算行内的比例（0-1）并映射到三个区域。如果计算有误，UI 会让人感觉不可预测。

The Conflict Scenario冲突场景

When a user moves files from folder A to folder B, some filenames may already exist in folder B. The system must detect these collisions before executing the move and present the user with resolution options.当用户将文件从文件夹 A 移动到文件夹 B 时，某些文件名可能已存在于文件夹 B 中。系统必须在执行移动之前检测这些冲突，并向用户呈现解决选项。

MoveConflictModal移动冲突弹窗

The modal presents each conflicting file with three resolution options per file: Overwrite (replace existing), Skip (leave it), or Rename (append a suffix like -copy-1). A global "Apply to all" checkbox streamlines bulk operations.弹窗为每个冲突文件提供三种解决选项：覆盖（替换已有文件）、跳过（保留原文件）或重命名（添加 -copy-1 后缀）。全局"应用到全部"复选框简化批量操作。

type ConflictResolution = 'overwrite' | 'skip' | 'rename';

interface ConflictItem {
  sourceFile: FileNode;
  targetExisting: FileNode;
  resolution: ConflictResolution;
}

// User resolves each conflict individually or uses "Apply to All"
const handleResolve = (id: string, resolution: ConflictResolution) => {
  setConflicts(prev =>
    applyToAll
      ? prev.map(c => ({ ...c, resolution }))
      : prev.map(c => c.sourceFile.id === id ? { ...c, resolution } : c)
  );
};

Partial Failure Rollback部分失败回滚

A move operation of 10 files might succeed for 7 and fail for 3. The system must handle this partial failure gracefully:移动 10 个文件的操作可能有 7 个成功、3 个失败。系统必须优雅地处理部分失败：

  1. Before the move batch starts, snapshot the state of all involved files and their positions.
  2. Execute moves sequentially or in a batch API call.
  3. If any fail, rollback only the failed items: restore their original parent, position, and undo any overwrites by restoring the replaced file.
  4. Show a summary: "7 files moved, 3 failed -- originals restored."
async function executeMoveWithRollback(items: MoveItem[], targetId: string) {
  const snapshots = items.map(item => ({
    item,
    originalParent: item.parentId,
    originalIndex: item.sortIndex,
    replacedFile: findExistingFile(targetId, item.name),
  }));

  const results = await batchMove(items, targetId);

  const failed = results.filter(r => !r.success);
  if (failed.length > 0) {
    for (const f of failed) {
      const snap = snapshots.find(s => s.item.id === f.id);
      if (snap) {
        restoreToParent(snap.item, snap.originalParent, snap.originalIndex);
        if (snap.replacedFile) restoreFile(snap.replacedFile);
      }
    }
  }
}

Optimistic UI with Server Reconciliation乐观更新与服务端校正

The UI moves files immediately (optimistic update). When the server response arrives, if there is a mismatch, the store reconciles by diffing the server state with the optimistic state and applying corrections. This keeps the UI feeling instant while maintaining data integrity.UI 立即移动文件（乐观更新）。当服务端响应到达时，如果存在不匹配，store 通过对比服务端状态和乐观状态的差异来校正。这使 UI 感觉即时响应，同时保持数据完整性。

Key Insight 核心洞察

The "Stop" action in the conflict modal is not a simple cancel -- it must gracefully stop an in-progress batch. This means the move loop must check a cancellation flag between each operation and still perform rollback on any items that were already moved if the user wants a full undo.冲突弹窗中的"停止"操作不是简单的取消——它必须优雅地停止正在进行的批处理。这意味着移动循环必须在每个操作之间检查取消标志，并且如果用户需要完全撤销，还需要对已移动的项目执行回滚。

const validatePath = (path: string): ValidationResult => {

SSE Stream MonitoringSSE 流式监听

After publishing, an SSE connection monitors the file for changes. When the file's content hash changes, the publish button transitions from "Published" to "Needs Update" without user intervention.发布后，SSE 连接持续监听文件变化。当文件内容哈希发生变化时，发布按钮自动从"已发布"转为"需要更新"，无需用户操作。

const monitorChanges = (fileId: string, publishedHash: string) => {
  const eventSource = new EventSource(`/api/sse/file-watch/${fileId}`);

  eventSource.onmessage = (event) => {
    const { hash } = JSON.parse(event.data);
    if (hash !== publishedHash) {
      setPublishState('needs-update');
    }
  };

  return () => eventSource.close();
};

Flush Before Save保存前刷新输入

A subtle but critical issue: contenteditable elements (like Tiptap editors) may have pending input that has not been committed to the React state. Before triggering a publish, the system calls editor.commands.blur() followed by a microtask delay to ensure all pending input events are flushed, the state is updated, and the save API receives the complete content.一个微妙但关键的问题：contenteditable 元素（如 Tiptap 编辑器）可能有尚未提交到 React state 的待处理输入。在触发发布前，系统调用 editor.commands.blur() 并等待微任务延迟，确保所有待处理的输入事件被刷新、状态被更新、保存 API 接收到完整内容。

const publishWithFlush = async () => {
  // Force flush any pending contenteditable input
  editor?.commands.blur();
  await new Promise(r => setTimeout(r, 0)); // yield to microtask
  await saveFile();
  await publishFile();
};
Key Insight 核心洞察

The SSE-based change detection removes the need for polling. The server pushes a lightweight hash comparison event whenever the file's underlying data changes, whether through the current user's edits, a collaborator's changes, or an API import. This keeps the publish button status accurate without client-side timers.基于 SSE 的变更检测消除了轮询的需要。无论是当前用户的编辑、协作者的修改还是 API 导入导致的数据变化，服务器都会推送轻量级哈希比较事件。这使发布按钮状态保持准确，无需客户端定时器。

WebSocket Reconnect SafetyWebSocket 重连安全机制

Preventing reconnect storms with mutex, backoff, and offline detection通过互斥锁、退避策略和离线检测防止重连风暴

The Reconnect Problem重连问题

When a WebSocket connection drops (network change, server restart, laptop sleep/wake), multiple parts of the application may simultaneously detect the disconnection and attempt to reconnect. Without coordination, this creates a "reconnect storm" -- multiple WebSocket instances competing, causing duplicate messages and wasted connections.当 WebSocket 连接断开时（网络切换、服务器重启、笔记本休眠/唤醒），应用的多个部分可能同时检测到断连并尝试重连。若无协调机制，会造成"重连风暴"——多个 WebSocket 实例竞争，导致消息重复和连接浪费。

Mutex-Based Reconnection基于互斥锁的重连

A simple boolean flag (isReconnecting) acts as a mutex. Only the first caller to set the flag proceeds with reconnection. Others see the flag is already set and exit early.一个简单的布尔标志（isReconnecting）充当互斥锁。只有第一个设置标志的调用者继续执行重连。其他调用者发现标志已设置后直接返回。

class SafeWebSocket {
  private ws: WebSocket | null = null;
  private isReconnecting = false;
  private retryCount = 0;
  private maxRetries = 8;
  private retryTimer: number | null = null;

  connect(url: string) {
    this.ws = new WebSocket(url);
    this.ws.onopen = () => {
      this.retryCount = 0;
      this.isReconnecting = false;
      this.onStatusChange('connected');
    };
    this.ws.onclose = () => this.scheduleReconnect();
    this.ws.onerror = () => this.scheduleReconnect();
  }

  private scheduleReconnect() {
    // Mutex: prevent concurrent reconnect attempts
    if (this.isReconnecting) return;
    if (this.retryCount >= this.maxRetries) {
      this.onStatusChange('failed');
      return;
    }
    if (!navigator.onLine) {
      this.onStatusChange('offline');
      this.waitForOnline();
      return;
    }

    this.isReconnecting = true;
    this.onStatusChange('reconnecting');

    // Exponential backoff: 1s, 2s, 4s, 8s, 16s...
    const delay = Math.min(1000 * Math.pow(2, this.retryCount), 30000);
    this.retryTimer = window.setTimeout(() => {
      this.retryCount++;
      this.connect(this.url);
    }, delay);
  }

  private waitForOnline() {
    const handler = () => {
      window.removeEventListener('online', handler);
      this.scheduleReconnect();
    };
    window.addEventListener('online', handler);
  }

  destroy() {
    if (this.retryTimer) clearTimeout(this.retryTimer);
    this.ws?.close();
    this.ws = null;
  }
}

Cleanup on Unmount卸载时清理

When a component that owns a WebSocket connection unmounts, it must close the connection and clear all timers/listeners. Failing to do so causes memory leaks and ghost connections.当拥有 WebSocket 连接的组件卸载时，必须关闭连接并清除所有定时器/监听器。否则会导致内存泄漏和幽灵连接。

useEffect(() => {
  const socket = new SafeWebSocket();
  socket.connect(wsUrl);

  // Cleanup: close connection, clear timers, remove listeners
  return () => socket.destroy();
}, [wsUrl]);
Key Insight 核心洞察

The combination of navigator.onLine check + online event listener is essential. Without it, the client wastes retries when the network is completely unavailable. By waiting for the online event, the first reconnect attempt happens when the network is actually reachable, making it much more likely to succeed on the first try.navigator.onLine 检查 + online 事件监听器的组合至关重要。没有它，客户端在网络完全不可用时会浪费重试次数。通过等待 online 事件，第一次重连尝试发生在网络实际可达时，大大提高首次成功率。
