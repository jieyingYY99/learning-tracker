程序入口 — 一切从这里开始

什么是 main.go？

每个 Go 程序都有一个 main.go 文件，就像每栋大楼都有一个正门一样。程序启动时，Go 会找到 func main() 这个函数，然后从那里开始执行。

打个比方 🎬

想象你是餐厅经理，每天早上开门前要做一堆准备工作：开灯、检查冰箱、准备食材、告诉服务员今天的菜单……main.go 就是你的"开店清单"。

启动流程（简化版）

1 读取配置文件（我要用哪个数据库？端口号是多少？）
↓
2 初始化日志系统（出了问题怎么记录？）
↓
3 连接数据库 MySQL + Redis
↓
4 初始化外部服务（OpenAI、Google AI、图片处理等）
↓
5 初始化技能系统（AI 能做哪些事？）
↓
6 启动 HTTP 服务器（打开大门，开始接客！）
↓
7 设置优雅关闭（关门前把客人都招待完再走）

关键代码速览

Go func main() { // 第1步：读配置 cfg := config.Load() // 第2步：初始化日志 logger.Init(cfg.Log.Level) // 第3步：连接数据库 db := storage.Init(cfg.Database) // 第4步：初始化 AI 服务 openaiClient := openai.NewClient(cfg.OpenAI.APIKey) // 第5步：初始化技能 skillService := skill.NewService(cfg.Skills) // 第6步：启动 HTTP 服务器 (端口 8090) server := api.NewServer(cfg, db, skillService, ...) server.Run(":8090") }

这是简化版哦，真实代码更长，但核心逻辑就是这些步骤。

优雅关闭 — Graceful Shutdown

当我们要停止服务器的时候（比如部署新版本），不能直接拔电源！我们要：

  • 先告诉新来的客人"我们快关门了"（停止接新请求）
  • 等正在服务的客人都搞完（最多等 280 秒）
  • 清理掉所有正在进行的 AI 对话流
  • 然后才真正关闭

为什么是 280 秒？因为 Kubernetes 给容器 300 秒的关闭时间，280 秒比它稍微短一点，留点缓冲。

环境变量 — 配置的载体

配置的值从哪来？通过环境变量！就像你跟朋友说"密码在纸条上"，而不是直接把密码写在门上。

路由与 API — 前台接待员

什么是 API 路由？

当用户（或者前端页面）想做某件事情时，他们会发送一个 HTTP 请求到我们的服务器。路由器的工作就是根据请求的"地址"，把它转发给正确的处理函数。

打个比方 📮

路由器就像邮局分拣员：

  • 寄到"一楼/文件科" → 交给文件管理模块
  • 寄到"二楼/聊天室" → 交给 AI 对话模块
  • 寄到"三楼/技能部" → 交给技能系统

HTTP 方法速学

方法意思比喻
GET拿东西看看去图书馆借书
POST创建新东西寄一封新信
PUT修改已有的东西改作业
DELETE删掉东西扔垃圾

项目里的核心路由

路由表 // === 文件管理 === POST /api/folder/v1/dir // 创建文件夹 GET /api/folder/v1/dir // 列出文件夹内容 POST /api/folder/v1/file/upload // 上传文件 GET /api/folder/v1/file/:filepath // 下载文件 DELETE /api/folder/v1/dir // 删除文件/文件夹 // === AI 对话 === POST /api/folder/v1/conversation/chat // 💬 发消息给 AI（流式） GET /api/folder/v1/conversation/ // 列出所有对话 GET /api/folder/v1/conversation/:uid // 查看某个对话 // === 技能 === GET /api/skills // 查看可用技能 POST /api/skill/:skillID/execute // 执行某个技能 // === 健康检查 === GET /health // 我还活着吗？

Gin 框架 — 我们的 HTTP 管家

项目使用 Gin 框架来处理 HTTP 请求。Gin 是 Go 里最流行的 Web 框架，又快又简单。

Go // 创建路由器 router := gin.Default() // 注册路由：当有人 GET /health 时，调用 HandleHealth 函数 router.GET("/health", folder.HandleHealth) // 路由组：给一组路由加前缀 v1 := router.Group("/api/folder/v1") { v1.POST("/dir", folder.HandleMKDir) v1.GET("/dir", folder.HandleDir) }

数据库 — 程序的记忆

为什么需要数据库？

程序跑着跑着一关机，内存里的东西就全没了。数据库就是程序的"长期记忆"——把重要的东西存到硬盘上，下次开机还在。

打个比方 📓

想象你有两种记事方式：

  • MySQL = 你的笔记本（结构化的，按格子填写，适合存文件信息、用户资料、聊天记录）
  • Redis = 你的便利贴（快速、临时的，适合存"谁正在线"、"这个对话正在进行中"等瞬时信息）

GORM — 数据库翻译官

直接写 SQL 太麻烦了，我们用 GORM（Go 的 ORM 框架）来跟数据库交流。它能把 Go 的结构体自动翻译成数据库的表。

Go // 定义一个 Go 结构体 type File struct { ID uint // 自动生成的唯一编号 UserID uint64 // 这个文件属于谁 Path string // 文件路径，比如 "/notes/日记.md" IsDir bool // 是文件还是文件夹？ Content string // 文件内容 CreatedAt time.Time // 什么时候创建的 } // GORM 会自动创建对应的 MySQL 表！ // 查询文件就像这样简单： var file File db.Where("user_id = ? AND path = ?", 123, "/notes/日记.md").First(&file)

软删除 — 后悔药

注意到 File 表有个 DeletedAt 字段吗？这叫软删除。用户删文件时，我们不是真的删掉，而是标记一个删除时间。这样：

  • 用户可以去"垃圾桶"恢复文件
  • 查询时自动过滤掉已删除的文件
  • 真正删除要等一段时间后才执行

C 它们功能一样，互为备份
D MySQL 是主数据库，Redis 只在 MySQL 挂了时用
06

AI 对话系统 — 项目的灵魂

这是项目最酷的部分！

用户在前端打字发消息，后端就要：接收消息 → 发给 AI 模型 → 把 AI 的回复一点一点"流"回给用户。

为什么是"流"？因为 AI 生成文字是一个一个字蹦出来的（就像 ChatGPT 一样），我们不能等它全部生成完再发送，那太慢了！

打个比方 📻

传统 API 像写信——你写完一整封信才寄出去。
流式响应（SSE）像打电话——你说一个字对方就听到一个字，实时的！

一条消息的完整旅程

1 用户发消息："帮我写一首诗"
↓
2 后端收到 POST /conversation/chat 请求
↓
3 把用户消息存到 MySQL
↓
4 组装上下文（之前的聊天记录 + 系统提示词）
↓
5 通过 LiteLLM 网关发给 AI 模型（Claude / GPT / Gemini）
↓
6 AI 一个字一个字地回复，通过 SSE 实时推给前端
↓
7 AI 回复完成，把完整回复存到 MySQL
↓
8 如果对话太长了，触发压缩（三层压缩架构）

三层压缩 — 无限记忆的秘密

聊天记录越来越多怎么办？全部发给 AI 太贵了（AI 按字数收费），而且它也处理不了那么多。所以我们设计了三层压缩：

对话压缩架构
Recent 层
最近的消息，完整保留
比如最近 20 条
↓ 旧消息被压缩
Summary 层
较早的消息，压缩成摘要
"用户之前讨论了 XX 话题"
↓ 更早的继续压缩
Archive 层
很久以前的消息，极度压缩
"历史上讨论过的主题列表"

这样 AI 既能"记住"以前聊过什么，又不会因为数据太多而变慢或太贵。

LiteLLM — AI 模型的"翻译中心"

我们不只用一个 AI，而是可以用 Claude、GPT、Gemini 等多个模型。LiteLLM 是一个统一的网关，让我们用同一套代码调用不同的 AI：

简化示意 // 不管用哪个 AI，代码都一样： response := litellm.ChatCompletion({ Model: "claude-sonnet-4-6-gcp", // 换成 "gpt-4o" 也行 Messages: chatHistory, Stream: true, })
小测验 #6
在三层压缩架构中，"Recent 层"保存的是什么？
A 所有历史消息的压缩摘要
B 用户的个人资料和偏好设置
C 最近的完整消息（原封不动保留的）

07

文件管理 — 云端的文件柜

文件存在哪里？

用户的文件信息（名字、路径、所有者）存在 MySQL 里，而文件内容本身存在 Google Cloud Storage（GCS）——一个超大的云端硬盘。

打个比方 🏢

MySQL 像图书馆的目录卡片（书名、作者、放在哪个架子上），GCS 像书架本身（真正放书的地方）。你先查目录找到位置，再去书架拿书。

文件上传流程

1 用户选择文件，点"上传"
↓
2 后端接收文件，计算 SHA256 哈希值（文件的"指纹"）
↓
3 在 MySQL 中创建文件记录（路径、大小、所有者等）
↓
4 把文件内容上传到 GCS
↓
5 如果是图片，生成缩略图
↓
6 返回文件 ID 给前端

版本控制 — 时光机

每次修改文件，旧版本不会消失，而是被保存到 FileVersion 表里。用户可以：

  • 查看文件的修改历史
  • 恢复到之前的某个版本
  • 把旧版本另存为新文件

就像 Google Docs 的"版本历史"功能！

FUSE 和 SharedFS — 让 AI 也能读写文件

有个很酷的功能：AI 助手可以像操作真实文件一样操作用户的文件！

  • SharedFS — 给 AI 助手提供读写用户文件的 API
  • FUSE — 把远程文件"挂载"到本地，像读本地文件一样读远程文件

安全靠 JWT Token：临时的"通行证"，用完就失效（默认 30 分钟）。

用户上传了一张图片，这张图片的"像素内容"存在哪里？
A MySQL 数据库的 File 表里
B Redis 缓存里

什么是技能（Skill）？

AI 不是万能的，但我们可以给它"装技能包"！每个技能就像一个专门的工具，让 AI 能做特定的事情。

打个比方 🎮

就像 RPG 游戏里的技能树！你的角色可以学"火球术"、"治愈术"、"隐身术"……每个技能有不同的效果。AI 也一样，它可以有"搜索网页"、"生成图片"、"翻译文档"等不同技能。

特点V1（文件系统）V2（数据库）

中间件 — 安保与管家

什么是中间件（Middleware）？

每个请求到达真正的处理函数之前，都要先经过一系列"检查站"。这些检查站就是中间件。

打个比方 🏨

你去五星级酒店：

  • 门口保安（CORS 中间件）→ 检查你是不是从允许的地方来的
  • 前台登记（请求日志中间件）→ 给你发个访客牌，记录你什么时候来的
  • 房卡验证（认证中间件）→ 你是住客吗？有权限进这个房间吗？

这些步骤都在你进入房间（处理函数）之前完成。

项目中的三大中间件

1. 请求 ID + 上下文提取

每个请求进来都分配一个唯一 ID（UUID），同时从请求头里提取用户信息：

Go // 中间件会从请求头里提取这些信息： UserID // 哪个用户发的请求 Browser // 用什么浏览器 Device // 什么设备 Country // 从哪个国家来的 RequestID // 这个请求的唯一编号

2. 请求/响应日志

记录每个请求的方法、路径、状态码、耗时。出了问题就能查"案发现场"。

3. CORS — 跨域保护

浏览器有个安全机制：A 网站不能随便访问 B 网站的 API。CORS 中间件决定"谁被允许来访问"：

Go // 本地开发 & 测试环境：谁都可以来 if env == "local" || env == "staging" { AllowAllOrigins: true } // 生产环境：只有 kuse.ai 可以来 if env == "prd" { AllowOrigins: ["https://kuse.ai", "https://*.kuse.ai"] }
小测验 #9
为什么生产环境的 CORS 不允许所有来源（AllowAllOrigins）？
A 因为允许所有来源会让服务器变慢
B 安全原因——防止其他恶意网站冒充用户访问我们的 API
C 因为 Go 语言在生产环境有这个限制
D 因为生产环境的网络带宽不够
10

部署上线 — 从代码到用户

代码写完了，怎么给用户用？

这是从"笔记本上的代码"变成"全世界都能访问的服务"的过程。

打个比方 🍕

写代码像做披萨，部署像送外卖：

  • Docker = 把披萨装进统一的外卖盒（标准化打包）
  • Container Registry = 外卖站点（存放打包好的东西）
  • Kubernetes = 外卖调度系统（决定哪个骑手送哪单）
  • CI/CD = 自动下单系统（代码一改，自动重新打包和配送）

Docker — 标准化打包

每台电脑的环境不一样（操作系统、软件版本……），Docker 的作用是把代码和它需要的一切都打包成一个"容器"，在任何机器上都能跑。

Dockerfile # 第一步：在一个干净的环境里编译 Go 代码 FROM golang:1.25 AS builder COPY . . RUN go build -o folder cmd/server/main.go # 第二步：把编译好的程序放进一个超轻量的镜像 FROM alpine:latest COPY --from=builder folder /app/folder CMD ["/app/folder"]

两步构建：先编译，再打包。最终的镜像只包含运行所需的东西，特别小。

CI/CD 自动化流水线

每次代码推送到 GitHub 的 main 分支，自动触发：

1 检测哪些文件改了（只重新构建改动的部分）
↓
2 Docker 构建镜像
↓
3 推送到 Google Container Registry
↓
4 kubectl 命令更新 Kubernetes 部署
↓
5 Kubernetes 滚动更新（先启动新的，再关掉旧的）

Kubernetes — 容器编排大师

当你有很多容器要管理时，Kubernetes（简称 K8s）就是你的总管：

  • 如果一个容器挂了，自动重启一个新的
  • 流量太大时，自动多开几个容器分担
  • 更新代码时，先启动新版本，确认没问题再关掉旧版本（滚动更新）
