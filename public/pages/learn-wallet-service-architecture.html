系統全景圖

兩個 Repo，一個錢包——理解它們的分工

Python kuse_ai（BFF 層）

  • 角色：Backend-For-Frontend，面向前端
  • 框架：FastAPI + Ormar ORM
  • 職責：用戶認證、路由、DTO 轉換、聚合回應
  • 錢包邏輯：轉發請求到 Go Wallet Service
  • 額外：Stripe 支付、禮品碼、每日獎勵

Go kuse-ai-go/wallet（核心服務）

  • 角色：Wallet 微服務，積分帳本
  • 框架：Gin + 原生 database/sql
  • 職責：積分增減、餘額查詢、消費記錄
  • 異步：Google Cloud Pub/Sub 消費者
  • 端口：:8089（HTTP）+ :6061（pprof）

整體請求流向

FLOW ┌──────────────┐ │ Web Client │ └──────┬───────┘ │ HTTP ┌──────▼───────────────────────────────┐ │ Python BFF (kuse_ai) │ │ Auth Middleware → Credit Views │ │ → credits_sys service (HTTP client) │ └──────┬───────────────────────────────┘ │ HTTP (wallet:8089) ┌──────▼───────────────────────────────┐ │ Go Wallet Service (kuse-ai-go) │ │ Handlers → Service → Repository │ │ ← MySQL (credits, consume_history) │ └──────────────────────────────────────┘ │ Pub/Sub (async) ┌──────▼───────────────────────────────┐ │ KSStream Consumer (Worker Pool) │ │ Token 計費 → FIFO 扣款 → 記錄歷史 │ └──────────────────────────────────────┘

Go // FIFO 扣款核心邏輯 — executeConsumeCredits() func executeConsumeCredits(tx, userID, amount) { // 1. SELECT ... FOR UPDATE 鎖定 credits := tx.Query("SELECT * FROM credits WHERE user_id=? AND status='active' AND remaining_amount > 0 ORDER BY expires_at ASC FOR UPDATE", userID) // 2. 依過期時間先後扣款 remaining := amount for _, c := range credits { deduct := min(remaining, c.RemainingAmount) c.RemainingAmount -= deduct remaining -= deduct if remaining == 0 { break } } // 3. 寫入 consume_history insertConsumeHistory(tx, userID, amount) }

鎖策略：悲觀鎖（FOR UPDATE）+ 樂觀鎖（WHERE remaining = expected）雙模式。

API 端點一覽

Go Wallet Service（:8089）的完整 API

核心錢包端點

GET /api/wallet/v1/balance 查詢餘額
Headers：X-User-ID（必填）, X-Tenant-Id（選填）
回應：total_amount、credit_details（按類型分組，含過期時間）
特點：含訂閱續約/結束日期資訊
POST /api/wallet/v1/award 發放積分
Payload：user_id/user_email, amount, reason, expires_at, tenant_id
用途：管理員手動發放積分，支援設定過期時間
POST /api/wallet/v1/revoke 撤銷積分
Payload：credit_ids 陣列
用途：將指定積分標記為 revoked 狀態
GET /api/wallet/v1/credit_history 積分歷史
Query：start_time, end_time, page_num, page_size
回應：分頁的積分增加記錄
GET /api/wallet/v1/consumption_history 消費歷史
Query：start_time, end_time, page_num, page_size
回應：分頁消費記錄，含對話標題、prompt、退款狀態
GET /api/wallet/v1/download_consumption_history CSV 匯出
回應：CSV 串流下載，支援 tenant owner 存取控制
GET /api/wallet/v1/subscription_credits 訂閱積分映射
Query：price_ids（陣列）
回應：Stripe price ID → 對應積分數量

OpenClaw 計費端點 Bearer Token

POST /api/wallet/v1/precheck 消費預檢
用途：模型執行前檢查用戶是否有足夠積分
Payload：user_id, tenant_id, request_id, conversation_uid, execution_kind

Token 計費邏輯

50+ 模型，一個公式

計費公式

FORMULA Credits = (input_tokens × input_price_per_M + output_tokens × output_price_per_M + cache_read × cache_read_price_per_M + cache_write × cache_write_price_per_M) ÷ 1,000,000 × 1.25 // 毛利率 (gross margin) × 904.523 // 積分/美元 (18000 credits ÷ $19.9)

FIFO 扣款策略

最快過期的先扣

按 expires_at ASC 排序，先消耗即將過期的積分。

悲觀鎖保護

SELECT ... FOR UPDATE 鎖定用戶所有 active 積分記錄，防止並發扣款。

逐筆扣減

遍歷積分記錄，每筆扣到 0 後移到下一筆，直到扣完所需金額。

部分扣款

如果餘額不足，actual_amount 可能小於 requested amount，仍然記錄消費。

冪等性保障

billing_adapter_requests 表通過 UNIQUE(scope, request_id) 確保同一筆 token-usage 不會被重複扣款。流程：

  1. 收到 token-usage 請求 → 嘗試插入 billing_adapter_requests
  2. 如果已存在（重複請求）→ 直接返回成功
  3. 如果是新請求 → 標記 pending → 發布 Pub/Sub → 標記 published
  4. 失敗 → 標記 failed + last_error

Q2：積分扣款時，先扣哪一筆？

A. 金額最大的先扣
B. 最近新增的先扣
C. 最快過期的先扣（expires_at ASC）
D. 隨機扣除
正確！FIFO 策略按 expires_at 升序排列，先消耗即將過期的積分，避免浪費。
提示：想想什麼策略最能避免積分浪費？即將過期的應該優先使用。

Q3：Python BFF 和 Go Wallet Service 之間怎麼通訊？

A. gRPC
B. HTTP（wallet:8089）
C. 共享資料庫
D. Message Queue

看看 credits_sys.py 裡的 WALLET_SERVICE_URL 是什麼？

Q4：billing_adapter_requests 表的主要用途是？

A. 儲存用戶的計費地址
B. 記錄 Stripe 支付記錄
C. 冪等性控制，防止重複扣款
D. 儲存計費方案配置
正確！UNIQUE(scope, request_id) 確保同一筆 token-usage 請求不會被重複處理。
這張表名叫 billing_adapter_"requests"——它關注的是請求的去重和狀態追蹤。
