Core Insight 核心洞察

Every store in this codebase started as a simple create() call. The pattern is not "choose the right middleware upfront" — it's "start simple, add middleware when pain appears."

项目中每个 store 都从简单的 create() 调用开始。模式不是"提前选对中间件"，而是"从简单开始，在痛点出现时添加中间件"。

The 4 Maturity Tiers 四层成熟度

Each tier represents a natural evolution point in store complexity
每层代表 store 复杂度的自然演进点

Tier 1

Minimal — Plain create() 最简模式

No middleware. State + setters. Perfect for UI flags, simple data holders.

无中间件。状态+setter。适合 UI 标志位和简单数据容器。

Examples 例子： useProStore (100 lines), useUndoStore (77 lines), useOutOfCreditsStore

// store/pro/index.ts — Tier 1: No middleware
// store/pro/index.ts — 第一层：无中间件

export const useProStore = create<ProPlanState>((set, get) => ({
  subscription: null,
  isLoading: false,

  setSubscription: (sub) => set({ subscription: sub }),
  fetchSubscription: async () => {
    set({ isLoading: true });
    const data = await api.getSubscription();
    set({ subscription: data, isLoading: false });
  },
}));
When to stay at Tier 1 何时保持第一层
  • Only 1-2 components consume this store 只有 1-2 个组件消费
  • No cross-store communication needed 不需要跨 store 通信
  • No async race conditions 无异步竞态
  • State is simple flags/objects 状态是简单标志/对象
Tier 2

Growth — +subscribeWithSelector OR +Optimistic 成长模式

First middleware or advanced pattern appears. Driven by re-render pain or UX needs.

出现第一个中间件或高级模式。由重渲染痛点或 UX 需求驱动。

Examples 例子： useSkillsStore (365 lines), useConnectorsStore (239 lines)

// store/skills/index.ts — Tier 2: Optimistic + per-item loading
// 第二层：乐观更新 + 逐项加载状态

export const useSkillsStore = create<SkillsState>((set, get) => ({
  skills: [],
  skillUpdating: {} as Record<string, boolean>,  // per-item loading
  skillErrors: {} as Record<string, string | null>,

  toggleSkill: async (skillId, enabled) => {
    // 1. Optimistic update 乐观更新
    set({ skills: updateSkillInList(skillId, enabled) });

    try {
      await api.updateSkill(skillId, { enabled });
    } catch {
      // 2. Revert on failure 失败时回滚
      set({ skills: updateSkillInList(skillId, !enabled) });
    }
  },
}));
Upgrade triggers from Tier 1 → Tier 2 升级触发条件
  • Components re-render too often (add subscribeWithSelector) 组件重渲染过多
  • User actions feel slow (add optimistic updates) 操作感觉慢
  • Need per-item loading states 需要逐项加载状态
  • Toggle/enable/disable patterns 切换启用/禁用模式
Tier 3

Mature — +Event Bus +AbortController +Named Selectors 成熟模式

Cross-component communication, request cancellation, fine-grained subscriptions.

跨组件通信、请求取消、细粒度订阅。

Examples 例子： useTrashStore (379 lines), useTurboTasksStore (217 lines)

// store/trash/index.ts — Tier 3: Full middleware + patterns

import { subscribeWithSelector } from 'zustand/middleware';

let _loadAbortController: AbortController | null = null;

const getInitialState = () => ({
  items: [] as TrashEntry[],
  isLoading: false,
  selectedIds: new Set<number>(),
  // ... all initial values
});

export const useTrashStore = create<TrashState>()(
  subscribeWithSelector((set, get) => ({
    ...getInitialState(),

    loadItems: async () => {
      // AbortController — cancel stale requests
      if (_loadAbortController) _loadAbortController.abort();
      const ctrl = new AbortController();
      _loadAbortController = ctrl;

      const items = await trashAPI.listTrash({ signal: ctrl.signal });
      if (ctrl.signal.aborted) return; // ignore stale
      set({ items });
    },

    reset: () => set(getInitialState()),
  }))
);

// Named selector exports — fine-grained subscriptions
export const useTrashItems = () => useTrashStore(s => s.items);
export const useTrashLoading = () => useTrashStore(s => s.isLoading);
export const useTrashSelectedIds = () => useTrashStore(s => s.selectedIds);

// Event bus — cross-store communication
subscribeToStoreEvent(STORE_EVENTS.ITEMS_MOVED_TO_TRASH, () => {
  if (useUIStore.getState().isTrashView) {
    useTrashStore.getState().loadItems();
  }
});
Upgrade triggers from Tier 2 → Tier 3 升级触发条件
  • Other stores need to react to this store's changes (add event bus) 其他 store 需要响应变化
  • Rapid navigation causes stale data (add AbortController) 快速导航导致过期数据
  • Many components subscribe but only use 1 field (add named selectors) 组件只用 1 个字段
  • Need clean reset for view switches (add getInitialState) 视图切换需要清洁重置
Tier 4

Enterprise — +devtools +Backup/Restore +Complex Trees 企业模式

Multiple middleware layers, tree data structures, backup/restore for mode switching.

多层中间件、树形数据结构、模式切换的备份/恢复。

Examples 例子： useAssetsStore (2837 lines), useCanvasStore (1173 lines)

// store/assets/index.ts — Tier 4: Enterprise patterns

export const useAssetsStore = create<AssetsState>()(
  subscribeWithSelector(
    devtools((set, get) => ({
      nodesByPath: {} as Record<string, Node>,
      childrenByPath: {} as Record<string, string[]>,
      onboardingBackup: null as OnboardingBackup | null,

      // Backup/Restore for onboarding mode
      enterOnboardingMockProject: async () => {
        const backup = {
          nodesByPath: cloneNodesMap(get().nodesByPath),
          childrenByPath: cloneChildrenMap(get().childrenByPath),
        };
        set({ onboardingBackup: backup, /* load mock data */ });
      },

      exitOnboardingMode: () => {
        const { onboardingBackup } = get();
        if (onboardingBackup) {
          set({ ...onboardingBackup, isOnboardingMode: false });
        }
      },
    }), { name: 'AssetsStore' })
  )
);
Upgrade triggers from Tier 3 → Tier 4 升级触发条件
  • Need Redux DevTools for debugging complex state (add devtools) 需要 DevTools 调试
  • Mode switching that must preserve/restore state (add backup/restore) 模式切换需保存恢复
  • Tree/graph data structures with parent-child relationships 树/图数据结构
  • Store exceeds 500 lines with multiple interconnected actions 超过 500 行

6 Key Patterns 六个核心模式

Each pattern solves a specific pain point. Learn when to add each one.
每个模式解决一个特定痛点。学习何时添加。

1. subscribeWithSelector Tier 2+

Pain solved 解决的痛点： Components re-render on ANY store change, even fields they don't use.

组件在任何 store 变化时重渲染，即使不使用相关字段。

// Without: component re-renders when ANY field changes
const { items } = useStore(); // re-renders on isLoading change too!

// With subscribeWithSelector: only re-renders when items change
const items = useStore(s => s.items); // uses Object.is comparison

// External subscription (non-React code):
useStore.subscribe(
  s => s.items.length,           // selector
  (len) => console.log(`${len} items`), // listener
  { equalityFn: shallow }        // optional comparator
);

Adoption 采用情况： 13/16 stores (81%) — the most common middleware.

2. Event Bus (StoreEventManager) Tier 3+

Pain solved 解决的痛点： Store A changes → Store B needs to react, but direct import creates circular deps.

Store A 变化 → Store B 需要响应，但直接 import 会造成循环依赖。

// Emitter (assets store):
emitStoreEvent(STORE_EVENTS.ITEMS_MOVED_TO_TRASH, { ids });

// Listener (trash store):
subscribeToStoreEvent(STORE_EVENTS.ITEMS_MOVED_TO_TRASH, () => {
  if (useUIStore.getState().isTrashView) {
    useTrashStore.getState().loadItems();
  }
});

Adoption 采用情况： 12/16 stores (75%) — replaces Redux-style middleware.

3. AbortController Tier 3+

Pain solved 解决的痛点： Rapid navigation/switching triggers multiple requests; stale response overwrites fresh data.

快速导航触发多个请求；过期响应覆盖新数据。

let _ctrl: AbortController | null = null;

loadItems: async () => {
  if (_ctrl) _ctrl.abort();         // cancel previous 取消前一个
  _ctrl = new AbortController();

  const data = await api.fetch({ signal: _ctrl.signal });
  if (_ctrl.signal.aborted) return; // discard stale 丢弃过期
  set({ items: data });
}

Adoption 采用情况： 2/16 stores (trash, assets) — only where rapid switching is common.

4. Named Selector Exports Tier 3+

Pain solved 解决的痛点： Components manually write selectors; inconsistent field access; easy to forget subscribeWithSelector.

// Without: every component writes its own selector
const items = useTrashStore(s => s.items);

// With named exports: standardized, discoverable
export const useTrashItems = () => useTrashStore(s => s.items);
export const useTrashLoading = () => useTrashStore(s => s.isLoading);

// Consumer: clean and simple
const items = useTrashItems();

Adoption 采用情况： 8/16 stores — trash (8 exports), filePreview, kuseClaw.

5. getInitialState / Reset Tier 3+

Pain solved 解决的痛点： View switch needs clean state, but manual reset misses fields. New fields added to store but not to reset logic.

const getInitialState = () => ({
  items: [],
  isLoading: false,
  error: null,
  selectedIds: new Set(),
});

// Reset preserving user preferences:
reset: () => {
  const { sortBy, sortOrder } = get(); // preserve 保留
  set({ ...getInitialState(), sortBy, sortOrder });
}

Adoption 采用情况： 4/16 stores — trash, share, onboarding, assets.

6. Optimistic Update + Revert Tier 2+

Pain solved 解决的痛点： API latency makes UI feel sluggish. User clicks toggle → waits 200ms+ → sees change.

toggleSkill: async (id, enabled) => {
  // 1. Optimistic: update UI immediately 即时更新 UI
  set({ skills: updateInList(id, enabled) });

  try {
    await api.update(id, { enabled });
  } catch {
    // 2. Revert on failure 失败时回滚
    set({ skills: updateInList(id, !enabled) });
    toast.error('Failed to update');
  }
}

Adoption 采用情况： 6/16 stores. Exception 例外： billing stores NEVER use optimistic — money operations always wait for server confirmation.

Quiz 测验

Scenario-based questions to test your upgrade judgment
基于场景的题目，测试你的升级判断力

1. You have a simple toggle store (50 lines). 3 components use it, but only 1 field each. Components re-render on every toggle even when their field didn't change. What do you add?
你有一个简单的 toggle store（50 行）。3 个组件使用它，每个只用 1 个字段。任何 toggle 都导致所有组件重渲染。添加什么？
subscribeWithSelector enables selector-based subscriptions so each component only re-renders when its specific field changes. Event bus is for cross-store communication. AbortController is for request cancellation.
subscribeWithSelector 启用基于选择器的订阅，每个组件只在其字段变化时重渲染。
2. Your store handles file listing. Users rapidly click between folders, and sometimes old folder data appears after new folder data loaded. What pattern fixes this?
你的 store 处理文件列表。用户快速点击文件夹，有时旧文件夹数据在新数据加载后出现。什么模式解决？
AbortController cancels the in-flight request and the if (ctrl.signal.aborted) return guard discards stale responses. This is exactly the pattern in useTrashStore.loadItems().
AbortController 取消进行中的请求，signal.aborted 检查丢弃过期响应。这正是 useTrashStore.loadItems() 的模式。
3. You're building a billing downgrade feature. User clicks "Downgrade" and you need to update the UI. Should you use optimistic updates?
你在构建降级功能。用户点击"降级"后需要更新 UI。是否使用乐观更新？
Never use optimistic updates for money operations. The server is the source of truth for billing state. Use a pendingPaymentKey lock to prevent double-submit, but always wait for scheduled_change === true from the server before updating UI.
金钱操作绝不使用乐观更新。服务器是计费状态的唯一真相来源。
4. Assets store (2837 lines) deletes a file. Trash store needs to refresh its list. How should they communicate?
Assets store（2837 行）删除文件。Trash store 需要刷新列表。它们如何通信？
The event bus (StoreEventManager) provides loose coupling between stores. Direct imports create circular dependencies. This is exactly how the project handles it: emitStoreEvent(STORE_EVENTS.ITEMS_MOVED_TO_TRASH) in assets → subscribeToStoreEvent in trash.
事件总线（StoreEventManager）提供 store 间的松耦合。直接 import 会造成循环依赖。
5. You have a store at Tier 2 (optimistic updates, 200 lines). It works fine with 2 consuming components. A PM asks you to add subscribeWithSelector, event bus, and AbortController "for best practices." What do you do?
你有一个 Tier 2 store（乐观更新，200 行）。2 个消费组件工作良好。PM 要求加 subscribeWithSelector、event bus 和 AbortController"按最佳实践"。你怎么做？
Don't add middleware preemptively. The maturity model is descriptive, not prescriptive. Add patterns only when real pain points emerge (re-renders, stale data, cross-store needs). Premature middleware adds complexity without benefit. "Start simple, add when pain appears."
不要预防性添加中间件。成熟度模型是描述性的，不是规定性的。只在真实痛点出现时添加。

Flashcards 闪卡

Click to flip. English front, Chinese back.
点击翻转。正面英文，背面中文。

subscribeWithSelector

When to add this middleware?

当组件因无关字段变化而重渲染时添加。启用 useStore(s => s.field) 的精准订阅 + 外部 subscribe(selector, listener)。项目中 13/16 store 使用，是最常见的中间件。

StoreEventManager

Custom event bus for cross-store communication

自定义事件总线，替代 Redux 中间件进行跨 store 通信。emitStoreEvent() 发送，subscribeToStoreEvent() 监听。避免直接 import 导致循环依赖。12/16 store 集成。

AbortController
in Zustand

Request cancellation pattern

防止快速切换导致过期数据覆盖。模式：模块级 let _ctrl → 每次请求前 abort 旧的 → 创建新 controller → 请求完成后检查 signal.aborted。只有 trash 和 assets 使用——仅在快速切换常见时添加。

Named Selector
Exports

useTrashItems(), useTrashLoading()

标准化的字段访问函数导出。好处：(1) 消费者代码更简洁；(2) IDE 自动补全可发现；(3) 确保使用 subscribeWithSelector。需要先添加 subscribeWithSelector 中间件。trash 有 8 个导出。

getInitialState()

Clean reset pattern

将所有初始值提取到函数中。reset: () => set({ ...getInitialState(), sortBy }) — 可选择保留用户偏好。新增字段时自动包含在 reset 中，防止遗漏。4/16 store 使用。

Optimistic Update
Exception

When NOT to use optimistic?

金钱/订阅操作绝不使用乐观更新。服务器是唯一真相来源。用 pendingPaymentKey 锁 UI 防止双重提交，但始终等服务端确认 scheduled_change === true。6/16 store 使用乐观更新，但 billing 例外。

Middleware
Composition Order

subscribeWithSelector(devtools(...))

Zustand 中间件从外到内包裹：subscribeWithSelector(devtools(fn))。subscribeWithSelector 在最外层才能对所有 state 变化生效。devtools 在内层提供 Redux DevTools 集成。项目中只有 assets 和 dataStore 使用双层组合。

Start Simple
Add When Pain Appears

The #1 principle of store maturity

不要预防性添加中间件。成熟度模型是描述性的，不是规定性的。pro store 100 行无中间件完全够用；assets store 2837 行需要全套。添加时机：真实痛点出现时（重渲染、过期数据、跨 store 需求）。

Store Upgrade Checklist 升级清单

Use this when evaluating whether a store needs upgrading. Check only if the pain point exists.
评估 store 是否需要升级时使用此清单。仅在痛点存在时勾选。

Before Adding Any Middleware 添加任何中间件之前

  • Is there a real, observed problem? (not hypothetical) 是否有真实观察到的问题？
  • Does the store have > 2 consuming components? 是否有 > 2 个消费组件？
  • Is the store > 100 lines? 是否超过 100 行？

subscribeWithSelector Checklist

  • Components re-render on unrelated field changes 组件因无关字段重渲染
  • > 3 components subscribe to the store > 3 个组件订阅
  • External (non-React) code needs to subscribe 非 React 代码需要订阅

Event Bus Checklist

  • Another store needs to react to this store's actions 其他 store 需要响应
  • Direct import would create circular dependency 直接 import 会循环依赖
  • Multiple unrelated UI areas respond to the same event 多个无关 UI 区域响应同一事件

AbortController Checklist

  • Users rapidly trigger the same async action (navigation, search) 用户快速触发同一异步操作
  • Stale responses have been observed overwriting fresh data 过期响应覆盖新数据已被观察到
  • The async operation takes > 200ms on average 异步操作平均 > 200ms

Named Selectors Checklist

  • subscribeWithSelector is already added 已添加 subscribeWithSelector
  • > 5 components each write their own inline selector > 5 个组件各自写内联 selector
  • Team members frequently ask "which field do I subscribe to?" 团队经常问"订阅哪个字段？"

getInitialState Checklist

  • View switch/unmount needs clean state reset 视图切换需要干净重置
  • Manual reset has missed fields before (bug history) 手动重置曾遗漏字段
  • Store has > 5 state fields 状态字段 > 5 个

devtools Checklist

  • State is complex (nested objects, trees, maps) 状态复杂（嵌套对象、树、映射）
  • Debugging requires inspecting state history 调试需要检查状态历史
  • Store has > 500 lines 超过 500 行
