核心设计概念

🔒 乐观锁 vs 悲观锁

系统主要使用乐观锁：UPDATE 时附带 WHERE remaining=旧值。若 RowsAffected=0，说明被其他事务修改了，需要退避重试。

乐观锁
不加锁，事后检查
吞吐量更高
可能需要重试
vs
悲观锁
SELECT FOR UPDATE
保证一次成功
延迟更高

📈 FIFO 积分消费

积分按 expires_at ASC 排序消费。最快过期的积分优先使用，减少过期浪费。

积分 A: 100个, 3月30日到期 ← 优先使用
积分 B: 500个, 4月30日到期
积分 C: 200个, 5月30日到期 ← 最后使用

🔀 基于用户的消息分片

消息通过 FNV1a(userId) % workerCount 路由。同一用户的消息始终路由到同一个 Worker，消除并发修改冲突。

用户 123 → hash % 8 = Worker 3
用户 456 → hash % 8 = Worker 7
用户 789 → hash % 8 = Worker 3（串行处理）

🛡️ 幂等性保证

Pub/Sub 保证至少一次投递。三重去重机制防止重复处理：

1. consume_history.request_id（UNIQUE 约束）
2. billing_adapter_requests（scope+request_id）
3. refund_status: none → refunded（单向转换）

💲 Token 动态定价引擎

基于实际 LLM Token 用量的动态定价，附加毛利率乘数。

积分 = (tokens x 模型单价 / 1M)
       x 1.25（毛利率）
       x 904.52（积分/美元）

⚡ 异步 vs 同步设计

余额查询是同步的（即时读取）。积分消费是异步的（通过 Pub/Sub），这样前端在 AI 推理期间不需要等待计费完成。
