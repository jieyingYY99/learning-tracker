📖 第一章

為什麼需要「兩棟建築」？

想像你開了一間很酷的公司。你需要一個地方讓客人進來用你的產品， 還需要一個地方製造產品。這兩件事情完全不一樣，所以需要分開的地方！

🏠 🎨 🖼️ 💬

創意工作室 = kuse-app-v2

這是客人走進來的地方！他們在這裡可以： 畫畫（Canvas 白板）、聊天（AI Chat）、 管理檔案（上傳圖片、PDF）。 就像一間漂亮的咖啡廳，客人在這裡享受服務。

🏭 🤖 ⚙️ 📦

機器人工廠 = junior 系列

這是在後面的工廠！客人看不到它，但它負責： 製造 AI 機器人（開 Kubernetes Pod）、 教機器人做事（性格注入）、 管理機器人宿舍（租戶隔離）。

🍕 用披薩店來比喻

kuse-app-v2 就像披薩店的「前台」— 客人在這裡點餐、看菜單、拿披薩。 前台本身不會做披薩，它只是把訂單遞給廚房。

junior-api 就像「廚房管理員」— 他決定用哪個烤箱（Kubernetes Pod）， 確保每個烤箱有足夠的材料（CPU/記憶體），並且客人之間的披薩不會搞混（租戶隔離）。

junior-app 就像「大廚」— 他接到訂單後，安排製作流程（Provisioning 狀態機）， 跟廚房管理員溝通要用哪個烤箱，還負責結帳（Stripe 計費）。

🎨 第二章

創意工作室：kuse-app-v2

這是使用者直接互動的地方。它是一個 Next.js 應用程式， 負責畫面顯示和簡單的後端代理工作。

🔐 身份驗證（Auth）— 門口的警衛 TypeScript ▼
就像進入工作室前要刷門禁卡一樣！
  • 使用 自訂 Token 系統（取代了以前的 Auth0）
  • 透過 HTTP-only Cookie 儲存登入狀態，像是一張看不見的通行證
  • 支援 Google OAuth — 用 Google 帳號一鍵登入
  • Token 有 7 天有效期，過期要重新登入
// 每次打 API 都會自動帶上通行證 axios.interceptors.request.use((config) => { config.headers['Authorization'] = `Bearer ${token}` return config })
📡 API 代理（Proxy）— 前台的傳話員 Next.js ▼
工作室前台不會自己做事，它把客人的需求「轉傳」給真正的後端：
  • /api/pdf → 轉傳給 Browser API 生成 PDF
  • /api/screenshot → 轉傳給截圖服務
  • /api/proxy-image → 幫忙代理外部圖片（避免跨域問題）
  • /api/version → 回報目前版本號
重點：它自己沒有資料庫！所有資料都存在真正的後端 API 上。
🎭 V1 Gateway — 通往舊世界的秘密通道 ▼
V2 正在漸進式取代 V1，但有些舊功能還在 V1 上。 就像新餐廳開張了，但甜點還在隔壁舊餐廳做：
  • /boards/*、/p/*、/favorite → 轉到 V1 後端
  • 所有 /api/* 路由留在 V2 處理
  • 根據環境（DEV/STAGING/PROD）決定轉去哪個 V1 伺服器
🖌️ 前端功能 — 客人使用的工具 ▼
這些是使用者直接看到和使用的東西：
  • Canvas 白板（Konva.js）— 在上面拖拉、縮放、畫圖
  • AI 聊天（SSE 串流）— 跟 AI 即時對話
  • 檔案管理（Uppy + DnD Kit）— 上傳、拖曳、預覽各種檔案
  • 富文本編輯（Tiptap）— 像 Notion 一樣的文字編輯器
  • Zustand 狀態管理 — 6 個專門的 Store 管理不同資料
🏭 第三章

機器人工廠：junior-api + junior-app

這裡是真正有趣的地方！機器人工廠由兩個部門組成： 「基礎設施部門」（junior-api）和「營運部門」（junior-app）。

⚙️ 基礎設施部門：junior-api（Go 語言）

這就是「廚房管理員」— 他管理所有的烤箱（Kubernetes Pods）， 確保每個客人的機器人都有獨立的空間。

📦 每個客人都有自己的小房間（Namespace） Go K8s ▼
想像一棟公寓大樓，每個租戶都有自己的房間：
  • Namespace = 一間公寓（tenant-abc123）
  • PVC = 公寓裡的衣櫃（2GB 存儲空間）
  • Deployment = 住在裡面的機器人（2 CPU / 8GB 記憶體）
  • Service = 公寓門牌號碼（ClusterIP）
  • Ingress = 郵局地址（abc123.claw.staging.junior.so）
  • Network Policy = 公寓的安全規則（只能從特定入口進出）
// Go 程式碼：為新租戶建立 Kubernetes Namespace func allocateTenant(tenantID string) { // 1. 建立 namespace createNamespace("tenant-" + tenantID) // 2. 建立存儲空間 createPVC(tenantID, "2Gi") // 3. 部署 AI 機器人 createDeployment(tenantID, image) // 4. 建立網路入口 createIngress(tenantID) }
🔒 每個操作都有鎖（Per-Tenant Mutex） Go ▼
就像廁所的門一樣 — 一次只能一個人用！
  • 每個租戶有自己的「鎖」
  • 防止兩個請求同時修改同一個租戶的資源
  • 失敗時自動回滾（rollback），把公寓拆掉重來
📋 提供的 API 端點 ▼
  • POST /api/tenants/allocate — 建造新的機器人房間
  • POST /api/tenants/status — 查看機器人狀態
  • POST /api/tenants/setEnv — 更新機器人的設定
  • POST /api/tenants/redeploy — 重啟機器人
  • POST /api/tenants/delete — 拆掉房間
  • POST /api/admin/pool/status — 查看整棟大樓的狀況
所有 API 都需要 x-api-key 才能呼叫，就像工廠的員工證。

🧑‍🍳 營運部門：junior-app（Next.js + 自訂 Server）

這就是「大廚」— 他知道製作流程的每一步， 從接單到出餐，還負責跟客人溝通和收錢。

🔄 7 步驟 Provisioning 狀態機 — 製作流程 TypeScript ▼
就像做一個玩具機器人的步驟：
  1. 📋 queued — 訂單排隊中
  2. 📧 creating_google_account — 幫機器人辦 email
  3. 🏗️ creating_service — 叫 junior-api 建造房間
  4. 🎭 configuring_service — 注入機器人的「性格」
  5. 🌐 waiting_domain — 等網址準備好
  6. 💓 waiting_ready — 送出第一則訊息，確認機器人活著
  7. ✅ ready — 完成！機器人上線了！
// 性格注入：讓每個機器人都獨一無二！ const personality = { 'IDENTITY_MD': generateIdentity(agentName), 'SOUL_MD': generateSoul(traits), 'DAY1_PLAN_MD': generateDay1Plan(company), 'TOOLS_MD': generateTools(integrations), } // 把性格當作環境變數注入到 Pod 裡 await juniorApi.setEnv(tenantId, personality)
🔌 WebSocket 代理 — 即時聊天的秘密通道 ▼
就像翻譯員，在客人和機器人之間傳話：
  • 客人透過 /chat WebSocket 連線
  • server.js 驗證身份 → 解密 Gateway Token → 連到機器人的 Pod
  • 雙向即時傳遞訊息
  • 機器人掛了？自動觸發恢復機制（5 分鐘冷卻時間）
  • 還支援 /vnc — 可以看到機器人的「螢幕」！
💳 Stripe 計費 — 收銀台 ▼
  • 透過 Stripe Webhook 追蹤訂閱狀態
  • 管理信用額度（每個用戶預設 20,000 credits）
  • 信用不足時顯示升級提示
  • LiteLLM 整合 — 為每個租戶生成虛擬 API Key 追蹤用量
🗄️ MySQL 資料庫 — 工廠的檔案室 ▼
儲存所有重要的資料：
  • users — 誰是客人
  • tenant_data — 機器人的狀態、網址、加密 Token
  • session — 登入通行證
  • billing — 付費狀態
  • agent_pool — 機器人的個人資料
  • invite_code — 邀請碼
