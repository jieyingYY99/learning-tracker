Virtual Scroll + Tree Flattening
虚拟滚动 + 树形扁平化
FrontendW12

Migrate recursive DOM tree to flattened array + react-window virtual scrolling: useFlattenedTree hook recursively expands nested folder tree into FlatTreeItem[], paired with useIndexTree for O(1) lookup, solving DOM performance issues with large folder nodes.

将递归 DOM 树迁移为扁平化数组 + react-window 虚拟滚动：用 useFlattenedTree hook 将嵌套的文件夹树递归展开为 FlatTreeItem[]，配合 useIndexTree 实现 O(1) 查找，解决大量文件夹节点导致的 DOM 性能问题。

Key Points 要点

  • Why flatten 为什么扁平化: Virtual scrolling libraries (react-window) need a flat list with fixed row heights. Recursive DOM trees can't be virtualized.
    虚拟滚动库（react-window）需要固定行高的扁平列表。递归 DOM 树无法被虚拟化。
  • FlatTreeItem structure 结构: Each item stores: id, depth, hasChildren, isExpanded, parentPath — all info needed to render one row without tree traversal.
    每个条目存储：id、depth、hasChildren、isExpanded、parentPath — 渲染一行所需的全部信息，无需遍历树。
  • IndexTree for O(1) lookup: pathToIndex Map enables instant scroll-to and selection by path, instead of scanning the entire list.
    pathToIndex Map 实现按路径即时滚动和选择，无需扫描整个列表。
  • Incremental updates 增量更新: When expanding/collapsing, splice affected rows instead of rebuilding the entire flattened list — O(subtree) not O(n).
    展开/折叠时，splice 受影响的行而非重建整个扁平列表 — O(子树) 而非 O(n)。
// Conceptual: tree → flat
function flatten(node, depth, expanded) {
  const items = [{ id: node.id, depth, isExpanded: expanded.has(node.id) }];
  if (expanded.has(node.id) && node.children) {
    for (const child of node.children) {
      items.push(...flatten(child, depth + 1, expanded));
    }
  }
  return items;
}

Move Conflict Resolution & Rollback
文件移动的冲突解决与失败回滚
FrontendW12

Fault-tolerant file move design: MoveConflictModal lets users choose overwrite/skip/rename on conflict, auto-restore replaced and deleted files on partial failure, and correct handling of stop behavior.

文件移动操作的容错设计：冲突时弹出 MoveConflictModal 让用户选择覆盖/跳过/重命名，部分失败时自动恢复被替换和被删除的文件到原位，以及 stop 行为的正确处理。

Key Points 要点

  • Three strategies 三种策略: Overwrite (replace target), Skip (keep target, don't move), Rename (add suffix like " (1)"). User chooses per-conflict or "apply to all".
    覆盖（替换目标）、跳过（保留目标不移动）、重命名（添加后缀如 " (1)"）。用户可逐个选择或"全部应用"。
  • Rollback on failure 失败回滚: Track all mutations in a rollback queue. If any move fails mid-batch, reverse completed moves by restoring files from the queue.
    在回滚队列中跟踪所有变更。如果批量移动中任一操作失败，从队列恢复已完成的移动。
  • Stop behavior 停止行为: "Stop" means cancel remaining moves but keep already-completed ones — different from rollback which undoes everything.
    "停止"意味着取消剩余移动但保留已完成的 — 不同于回滚（撤销所有操作）。

WebSocket Reconnect Safety
WebSocket 重连安全机制
FrontendW12

Robust WebSocket connection design: offline detection (navigator.onLine + online/offline events), reconnect race condition prevention, exponential backoff retry, and retry status UI feedback.

WebSocket 连接的健壮性设计：离线检测（navigator.onLine + online/offline 事件）、重连时的竞态防护（防止多个重连同时进行）、指数退避重试、以及重试状态的 UI 反馈。

Key Points 要点

  • Race condition guard 竞态防护: Use a reconnecting flag + connection ID. If a new reconnect starts before the old one finishes, abort the old attempt.
    使用 reconnecting 标志 + 连接 ID。如果新的重连在旧的完成之前开始，中止旧的尝试。
  • Exponential backoff 指数退避: Retry delays: 1s → 2s → 4s → 8s → 16s → max 30s. Prevents server overload from aggressive reconnects.
    重试延迟：1s → 2s → 4s → 8s → 16s → 最大 30s。防止激进重连导致服务器过载。
  • Offline detection 离线检测: Listen to window 'online'/'offline' events. When offline, don't attempt reconnect — wait for 'online' event, then reconnect immediately.
    监听 window 的 'online'/'offline' 事件。离线时不尝试重连 — 等待 'online' 事件后立即重连。
  • UI feedback UI 反馈: Show reconnection status: "Connecting..." with attempt count, "Offline" badge, "Connected" success state.
    显示重连状态："连接中..." 含尝试次数、"离线"标记、"已连接"成功状态。
// Exponential backoff pattern
const getDelay = (attempt) => Math.min(1000 * Math.pow(2, attempt), 30000);

// Race condition guard
let connectionId = 0;
function reconnect() {
  const myId = ++connectionId;
  // ... connect ...
  if (myId !== connectionId) return; // Aborted by newer attempt
}

1. Why must a tree be flattened before using react-window for virtual scrolling?
为什么在使用 react-window 虚拟滚动前必须将树扁平化？
A) react-window doesn't support any nesting 不支持任何嵌套
B) Trees use too much memory 树占用太多内存
C) Virtual scrolling needs a flat list with fixed row heights to calculate which items are visible 虚拟滚动需要固定行高的扁平列表来计算哪些项可见
D) It's a React limitation React 的限制
Virtual scrolling works by calculating which rows are visible based on scroll position and fixed row height. A recursive DOM tree has dynamic heights and nested structures that prevent this calculation. Flattening converts the tree into a simple array where each item's position can be computed as index × rowHeight.
虚拟滚动通过滚动位置和固定行高计算可见行。递归 DOM 树有动态高度和嵌套结构，阻止了这种计算。扁平化将树转为简单数组，每项位置可通过 index × rowHeight 计算。
2. What does IndexTree's pathToIndex Map provide?
IndexTree 的 pathToIndex Map 提供了什么？
A) O(n) search through all nodes 遍历所有节点的 O(n) 搜索
B) O(1) lookup to find any node's index by its path 通过路径 O(1) 查找任何节点的索引
C) Automatic sorting of tree nodes 自动排序树节点
D) CSS styling data for each row 每行的 CSS 样式数据
The pathToIndex Map is built during flattening: each node's path is mapped to its index in the flat array. This enables O(1) operations like "scroll to this file" or "select this folder" without scanning the entire list.
pathToIndex Map 在扁平化时构建：每个节点的路径映射到其在扁平数组中的索引。这使"滚动到此文件"或"选择此文件夹"等操作为 O(1)，无需扫描整个列表。

B) CSS :hover pseudo-class with transition delay CSS :hover 伪类加过渡延迟

4. What is the difference between "Stop" and "Rollback" in move conflict resolution?

A) They are the same thing 它们是同一回事
B) Stop undoes all moves, Rollback keeps completed ones 停止撤销所有，回滚保留已完成的

"Stop" means: cancel all remaining pending moves, but don't undo the ones that already completed successfully. "Rollback" means: undo ALL changes, restoring every file to its original location. This distinction is important for user expectations — sometimes partial completion is acceptable.

6. How does WebSocket reconnect prevent race conditions when multiple reconnects fire?
WebSocket 重连如何防止多个重连同时触发时的竞态条件？
A) Using a mutex lock 使用互斥锁
B) Incrementing a connectionId — if the current id doesn't match, the older attempt aborts 递增 connectionId — 如果当前 id 不匹配，旧的尝试中止
C) Queuing all reconnect requests 将所有重连请求排队
D) Disabling reconnect after the first attempt 第一次尝试后禁用重连
A global connectionId counter is incremented each time reconnect() is called. The reconnect function captures its own ID at the start. After the async connection attempt, it checks if its ID still matches the global counter. If not, a newer reconnect has started, so this one aborts silently.
每次调用 reconnect() 时递增全局 connectionId 计数器。重连函数在开始时捕获自己的 ID。异步连接尝试后，检查其 ID 是否仍匹配全局计数器。如果不匹配，说明更新的重连已开始，当前静默中止。
7. What is the correct exponential backoff sequence for WebSocket reconnect?
WebSocket 重连的正确指数退避序列是什么？
A) 1s → 2s → 4s → 8s → 16s → max 30s
B) 1s → 1s → 1s → 1s (constant) 固定间隔
C) 5s → 10s → 20s → 40s → max 60s
D) 100ms → 500ms → 1s → 5s → max 10s
The formula is: Math.min(1000 * Math.pow(2, attempt), 30000). Starting at 1 second, each retry doubles the delay, capping at 30 seconds. This prevents overwhelming the server while still attempting reconnection reasonably quickly.
公式为：Math.min(1000 * Math.pow(2, attempt), 30000)。从 1 秒开始，每次重试加倍延迟，上限 30 秒。防止服务器过载的同时保持合理的重连速度。
