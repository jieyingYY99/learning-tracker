Layer 1 — StreamManager

Connection Control & Retry 连接控制与重试

Manages the SSE connection lifecycle: creates/destroys connections, handles retry with exponential backoff, supports 3 retry strategies (AUTO/MANUAL/RELOAD), and provides visibility-based reconnection.

管理 SSE 连接生命周期：创建/销毁连接、指数退避重试、3 种重试策略（AUTO/MANUAL/RELOAD）、页面可见性恢复重连。

// Key pattern: AsyncGenerator wrapping fetchEventSource
// 核心模式：AsyncGenerator 封装 fetchEventSource

private async *connectAndStream() {
  const eventQueue: RawSSEEvent[] = [];
  let resolveNext, isComplete = false;

  fetchEventSource(url, {
    onmessage: (event) => {
      const parsed = this.parseSSEEvent(event);
      if (resolveNext) {
        resolveNext({ value: parsed, done: false });
      } else {
        eventQueue.push(parsed);
      }
    },
    onclose: () => { isComplete = true; }
  });

  // Yield events as they arrive
  while (!isComplete || eventQueue.length > 0) {
    if (eventQueue.length > 0) {
      yield eventQueue.shift()!;
    } else {
      yield await new Promise(resolve => { resolveNext = resolve; });
    }
  }
}

Layer 3 — MessageBuilder

Incremental Message Construction 增量消息构建

Builds Message objects incrementally as events arrive. Merges consecutive text parts with the same eventId to avoid fragmentation. Supports 11 part types including tool_call with streaming content and ui_block for interactive cards.

随事件到达增量构建 Message 对象。基于 eventId 合并连续同类文本 Part 避免碎片化。支持 11 种 Part 类型，包含带流式内容的 tool_call 和交互卡片 ui_block。

type MessagePart.type =

  | 'thinking'          // AI 思考过程
  | 'coding'            // 代码块
  | 'image'             // 图片
  | 'table'             // 表格 (headers + rows)
  | 'tool_call'         // 工具调用（含 streamingContent[]）
  | 'selected_sources'  // 用户选中的文件/文件夹
  | 'agent_block'       // SubAgent 聚合块
  | 'editor_selection'  // 编辑器选区
  | 'highlights'        // 引用高亮
  | 'ui_block';         // 交互卡片 (ask_user, suggest_connectors)

// Merge strategy: same eventId → concatenate content
// 合并策略：相同 eventId → 拼接内容
appendText(content, type, eventId) {
  if (lastPart.type === type && lastPart.eventId === eventId) {
    lastPart.content += content;  // Merge 合并
  } else {
    this.parts.push({ type, content, eventId }); // New part 新建
  }
}

// Direct @microsoft/fetch-event-source usage
// 直接使用 @microsoft/fetch-event-source

import { fetchEventSource } from '@microsoft/fetch-event-source';

export async function refine(data, onMessage, onError) {
  const controller = new AbortController();
  const token = await getAccessToken();

  await fetchEventSource(url, {
    method: 'POST',
    headers: { Authorization: `Bearer ${token}`, 'X-Request-ID': uuidv4() },
    body: JSON.stringify(data),
    signal: controller.signal,
    onmessage(msg) {
      if (msg.event === 'error') { onError(msg.data); return; }
      onMessage(msg.data); // Single callback, no parsing layers
    },
    onerror(err) { throw err; } // No retry, just throw
  });
}

Characteristics 特点:

  • Single callback — no event classification 单一回调，无事件分类
  • No retry mechanism 无重试机制
  • No Agent/Tool lifecycle 无 Agent/Tool 生命周期
  • Uses @microsoft/fetch-event-source directly 直接使用第三方库

Decision Guide 决策指南

Does your SSE stream involve multiple agents or tool execution? 你的 SSE 流是否涉及多 Agent 或 Tool 执行?

YES → Does it need retry/reconnection support? 需要重试/重连?

YES → 3-Layer Architecture (StreamManager + Processor + Builder)

三层架构：连接管理 + 事件处理 + 消息构建

NO → 2-Layer: StreamManager + simple processor (skip MessageBuilder complexity)

两层方案：StreamManager + 简单 processor（跳过 MessageBuilder）

NO → Is it a one-shot AI request? 一次性 AI 请求?

YES → Simple SSE (direct fetchEventSource + single callback)

简单方案：直接 fetchEventSource + 单回调

NO → Evaluate: if text-only stream with retry needs → just StreamManager wrapper

评估：如果只需文本流 + 重试 → 只用 StreamManager 封装

Retry Strategy Detail 重试策略详解

Strategy 策略 When 触发条件 Behavior 行为
AUTO Network error, server 5xx 网络错误/服务端 5xx Exponential backoff (1s → 2s → 4s → 8s → 16s), max 5 retries. Reuses Request-ID + Last-Event-ID 指数退避，复用 ID 断点续传
MANUAL User clicks retry, or 4xx errors 用户点击重试/客户端 4xx New Request-ID, clear Last-Event-ID, create new user message 新 ID，重新发送
RELOAD Disconnected > 10 min (TTL expired) 断连超过 10 分钟 Reload entire conversation history, server cache expired 重载整个对话
Visibility Page becomes visible after being hidden 页面从隐藏变为可见 Auto-retry if network error exists, uses startAutoRetry() 自动重试网络错误

Quiz 测验

基于场景的决策测验 — 不只考"是什么"，而是考"你会怎么选"

Q1: You're building a new AI "refine text" feature. The AI returns streaming text, no tool calls, no agents. The request takes ~3 seconds. Which SSE approach would you use?

你在做一个新的 AI "润色文本"功能。AI 只返回流式文本，没有 Tool 调用，没有 Agent。请求约 3 秒。你会选哪种 SSE 方案?

Correct! For a simple, short-lived text stream with no agents or tools, the simple approach (like refine.ts) is sufficient. Adding 2500 lines of architecture for a 3-second text stream is overkill. 正确！对于简单的短时文本流，简单方案就够了。为 3 秒的文本流添加 2500 行架构是过度工程。

Q2: The SSE stream receives a text event with the same eventId as the previous text event. What does MessageBuilder do?

SSE 流收到一个与上一个 text 事件具有相同 eventId 的 text 事件。MessageBuilder 会怎么做?

Correct! MessageBuilder merges consecutive text events with the same eventId by concatenating content. This prevents message fragmentation — without it, each SSE chunk would create a separate Part. 正确！MessageBuilder 通过 eventId 合并连续的同类文本事件。这防止了消息碎片化。

Q3: The user's laptop goes to sleep for 12 minutes during a streaming response, then wakes up. What happens?

用户的笔记本在流式响应过程中休眠了 12 分钟，然后醒来。会发生什么?

Correct! ServerCacheTTL is 10 minutes. After 12 minutes of disconnection, the server cache has expired, so Last-Event-ID won't match anything. StreamManager calls onReloadRequired() to reload the entire conversation. 正确！服务端缓存 TTL 是 10 分钟。断连 12 分钟后服务端缓存过期，StreamManager 调用 onReloadRequired() 重载整个对话。

Q4: A SubAgent (e.g., "Word" agent) starts, creates files, and ends. Where are its messages stored during streaming?

一个 SubAgent（如 "Word" agent）启动、创建文件、结束。流式传输期间它的消息存储在哪里?

Correct! SubAgent messages are accumulated in subAgentMessages Map (keyed by agent_execution_id). When agent_end arrives, they're wrapped into an AgentBlock and placed into the main message via replaceAgentBlock(). 正确！SubAgent 消息累积在 subAgentMessages Map 中。agent_end 到达时，聚合为 AgentBlock 放入主消息。

Q5: After a network error during streaming, the user's page becomes visible again (within 5 minutes). What retry path is taken?

流式传输中发生网络错误后，用户的页面在 5 分钟内重新可见。会走哪条重试路径?

Correct! Within TTL (10 min), visibility retry uses startAutoRetry() which reuses Request-ID and Last-Event-ID for seamless resume. The server matches the cache and sends only events after the last one received. 正确！TTL 内（10 分钟），可见性重试用 startAutoRetry() 复用 ID 实现断点续传。
