1. NewTaskView Conversation History Integration
NewTaskView 对话历史集成

Land the ConversationHistory popup on the NewTaskView entry page with waterfall pagination. Reuses the same component as Chat but with currentConversation=null (no highlight) and the parent owns the fetch/optimistic-update logic.

把 Chat header 里的 ConversationHistory 组件搬到着陆页 NewTaskView，用瀑布流分页展示历史对话。currentConversation 传 null（着陆页没有“当前对话”的概念），由父组件 NewTaskView 自己管理 fetch、分页状态和 optimistic delete/rename。

Key Pattern 关键模式

// paginationOffsetRef + hasMoreRef: ref-based guard against closure staleness
const paginationOffsetRef = useRef(0);
const isLoadingMoreRef = useRef(false);
const hasMoreRef = useRef(true);

const loadMore = useCallback(async () => {
  if (isLoadingMoreRef.current || !hasMoreRef.current) return;
  isLoadingMoreRef.current = true;
  try {
    const res = await getFolderConversations({
      limit: 20, offset: paginationOffsetRef.current, mode: 'waterfall',
    });
    paginationOffsetRef.current += res.data.length;
    hasMoreRef.current = Boolean(res.has_more);
    setConversations(prev => dedupByUid([...prev, ...res.data]));
  } finally {
    isLoadingMoreRef.current = false;
  }
}, []);

// Delete with offset sync (keep pagination aligned)
const handleDelete = useCallback(async (id: string) => {
  await deleteFolderConversation(id);
  setConversations(prev => prev.filter(c => c.uid !== id));
  paginationOffsetRef.current = Math.max(0, paginationOffsetRef.current - 1);
}, []);

Comparison 对比

Same component, two usage contexts. Chat has a current conversation (highlight target), NewTaskView doesn't. Chat's parent already owns the list; NewTaskView's parent has to bootstrap fetch + pagination. The ref-based pagination guard is the novel part — it prevents the classic closure trap where loadMore reads a stale offset state.

同一个组件两种使用场景。Chat 有当前对话需要高亮，NewTaskView 没有。Chat 的父组件已有对话列表，NewTaskView 要自己 bootstrap 分页 fetch。ref-based 分页保护是新颖点 — 避免闭包捕获过时的 offset 状态。

Trade-offs 取舍

  • Ref dual-guard: reliable, but you must keep ref and state in sync （ref 双重保护可靠，但需要手动同步 ref 和 state）
  • Cancelled flag: simple (no AbortController), but fetch still completes (wasted bandwidth) （cancelled 标记简单，但 fetch 仍会完成，浪费带宽）
  • Optimistic update without rollback: faster UX, but silently fails if API errors （乐观更新无回滚 UX 快但 API 失败时会静默）

2. Background vs Item Context Menu
背景 vs 条目双上下文菜单

Finder now shows two different menus on right-click: one when you hit a file/folder, one when you hit empty space. The dispatcher uses closest('[data-finder-item]') to decide which menu to open, and automatically replaces selection if the right-clicked item isn't currently selected.

Finder 的右键现在有两种菜单：击中 item 显示条目菜单，击中空白显示背景菜单（New folder / Paste）。分派逻辑用 closest('[data-finder-item]') 做 DOM 探测，右键未选中条目时会自动先选中再显示菜单，符合 macOS/Windows 习惯。

Key Pattern 关键模式

// Ref captures current selection to avoid closure staleness
const selectedSetRef = useRef(selectedSet);
selectedSetRef.current = selectedSet;

const handleContextMenu = useCallback((e: React.MouseEvent) => {
  e.preventDefault();
  const { safeX, safeY } = clamp(e.clientX, e.clientY, 216, 200);

  const itemEl = (e.target as HTMLElement).closest('[data-finder-item]');
  if (itemEl) {
    const path = itemEl.getAttribute('data-finder-item')!;
    // Right-click on NOT-selected → replace selection with this item
    if (!selectedSetRef.current.has(path)) {
      selectMultiple([path], { skipChatSync: true });
    }
    setBgCtxMenu(m => m.open ? { ...m, open: false } : m);
    setCtxMenu({ open: true, x: safeX, y: safeY });
  } else {
    // Empty space → background menu
    setCtxMenu(m => m.open ? { ...m, open: false } : m);
    setBgCtxMenu({ open: true, x: safeX, y: safeY });
  }
}, [selectMultiple]);

// Outside-click handles BOTH menus via querySelectorAll
window.addEventListener('click', (e) => {
  const menus = document.querySelectorAll('[data-context-menu]');
  for (const menu of menus) if (menu.contains(e.target)) return;
  closeBothMenus();
}, true);

Comparison 对比

Key upgrade: from state-driven dispatch (selection.length) to DOM-driven dispatch (closest data-attribute). State-driven can't tell "empty space" apart from "nothing selected" — both have selection.length === 0. DOM探测 reads the actual click target, correctly distinguishing the two.

核心升级：从状态驱动分派（selection.length）改为 DOM 驱动分派（closest data-attribute）。状态驱动无法区分"空白区"和"没选中条目"，两者 selection.length 都是 0；DOM 探测读实际点击目标能正确区分。

Trade-offs 取舍

  • Dual state (ctxMenu + bgCtxMenu): more state to manage, but clean separation （两个 state 多一点负担，但语义清晰）
  • selectedSetRef is mandatory: without it, handleContextMenu captures stale selection （必须用 ref 避免闭包捕获过时的 selection）
  • querySelectorAll outside-click: one listener handles N menus （一个监听器处理多个菜单的外部点击）

3. localStorage View-State Persistence
localStorage 视图状态持久化

Zustand re-initializes on every page refresh. If a user was in folder view and refreshes, they'd jump back to NewTaskView unless we persist the flag. The pattern: a getInitialX() helper reads localStorage at store init, and resetAllViews writes the flag during view switches.

Zustand store 每次刷新都会重新初始化。如果用户在 folder view 刷新，会跳回 NewTaskView，除非把这个视图 flag 持久化到 localStorage。模式是：getInitialX() helper 在 store 初始化时读 localStorage，resetAllViews 在切换视图时写回。

Key Pattern 关键模式

// Tri-state init: null (first-time) vs 'true' vs 'false'
const getInitialNewTaskView = (): boolean => {
  if (typeof window === 'undefined') return true;  // SSR guard
  const saved = window.localStorage.getItem(NEW_TASK_VIEW_STORAGE_KEY);
  return saved === null ? true : saved === 'true';  // default true for first-time users
};

// Atomic write inside resetAllViews
resetAllViews: (merge, options) => {
  const state = get();
  if (!options?.keepPreview) useFilePreviewStore.getState().hide();
  if (merge?.isNewTaskView) {
    window.localStorage.setItem(NEW_TASK_VIEW_STORAGE_KEY, 'true');
  } else {
    window.localStorage.setItem(NEW_TASK_VIEW_STORAGE_KEY, 'false');
  }
  set({ isTrashView: false, isNewTaskView: false, ...merge });
},

Why NewTaskView defaults to true

First-time users should land on NewTaskView (the onboarding entry). After first login, the store reads null from localStorage → returns true. If we used the removeItem cleanup style like KuseClaw, exiting NewTaskView would delete the key, causing the next refresh to incorrectly read null → true again. Hence the explicit 'false' write.

NewTaskView 对首次用户默认开启（着陆页）。首次登录时 localStorage 返回 null，helper 视为 true。如果像 KuseClaw 那样用 removeItem 清理，退出 NewTaskView 会删除键，下次刷新又错误地视为 true。因此必须显式写 'false'。

SSR guard is mandatory 必须加 SSR 保护

Next.js renders stores on the server where window is undefined. Every getInitialX must check typeof window === 'undefined' first, or the build crashes.

4. TTL-Based Resource Ensure
TTL 周期性资源校验

The old ensureDefaultFolder used a one-shot flag — once created successfully, never check again. Bug: user deletes the default folder on another device, this client never re-creates it. Fix: 60s TTL, re-verify after expiry.

旧版 ensureDefaultFolder 用一次性 flag — 成功创建后永不重试。Bug 场景：用户在另一台设备删除默认文件夹，这台客户端永远不会重新创建。修复：60s TTL，过期后重新校验。

Key Pattern 关键模式

let lastEnsuredAt = 0;
const ENSURE_TTL = 60_000;

export async function ensureDefaultFolder() {
  if (Date.now() - lastEnsuredAt > ENSURE_TTL) {
    try {
      const { id } = await mkdir(DEFAULT_FOLDER.path);
      lastEnsuredAt = Date.now();
    } catch (e) { /* folder already exists is fine */ }
  }
  return DEFAULT_FOLDER;
}

Decision matrix 决策矩阵

"Create once, never changes" (session id) → one-shot flag "Should exist, may be deleted externally" → TTL (60s for user ops) "Needs fresh data" (counters) → polling / SSE "Explicit invalidation" (post-action) → cache-then-invalidate

Why 60s? 为什么是 60s

TTL is a consistency vs. call-cost tradeoff. For default folder: user-driven external mutations (deleting on another device) are rare and user-paced. A 60s staleness window is imperceptible to users. Much shorter (like 5s) would hammer the backend; much longer (like 1 hour) would frustrate users who just synced across devices.

TTL 本质是一致性 vs 调用成本的权衡。默认文件夹由用户操作触发外部变化（另一设备删除），频率低且节奏慢。60s 延迟用户无感；更短会打爆后端，更长会让刚跨设备同步的用户困惑。

5. Centralized View-Switch Side Effect
视图切换副作用集中化

Moved "hide file preview when switching to exclusive views" from individual callers (enterTrashView, enterSharedWithMeView, etc.) into resetAllViews itself, with an options.keepPreview opt-out. Added a no-op guard to hide() to avoid wasteful re-renders.

把"切换到互斥视图时关闭 file preview"这个副作用，从多个调用点（enterTrashView、enterSharedWithMeView 等）搬到 resetAllViews 内部，提供 options.keepPreview opt-out 豁免。配套给 hide() 加 no-op guard 避免无谓重渲染。

Key Pattern 关键模式

// In store/ui/index.ts
resetAllViews: (merge, options) => {
  const state = get();
  // No-op guard: nothing to do?
  if (!merge && !state.isTrashView && !state.isSharedWithMeView && ...) return;

  // Side effect centralized here, default-executed, opt-out via flag
  if (!options?.keepPreview) useFilePreviewStore.getState().hide();

  set({ isTrashView: false, ..., ...merge });
},

// In store/filePreview/store.ts — no-op guard
hide: () => {
  if (!get().isVisible) return;
  set({ isVisible: false, selectedFileId: null });
},
Deep Dive — 深度分析

3 side-effect coupling patterns

Manual-at-callsite (old)

enterTrashView() {
  resetAllViews({...});
  filePreviewStore.hide();  // easy to forget
}

Easy to miss when adding new views

Centralized opt-out (new)

resetAllViews(merge, options) {
  if (!options?.keepPreview)
    filePreviewStore.hide();
  // ...
}

Default-safe: forgot to opt-out = correct behavior

Opt-in vs Opt-out is a safety stance

Opt-in: you must say "yes, hide preview" each time → forgetting keeps preview open → potentially wrong.
Opt-out: default is hide, you must say "no, keep preview" when needed → forgetting hides preview → safer default.

Opt-in：每次要明确说"是的，关闭预览"，漏写就保留预览 → 错误偏向；
Opt-out：默认关闭，需要时明确说"保留预览"，漏写就关闭 → 安全偏向。

Why the no-op guard matters

With subscribeWithSelector, any set() call triggers subscribers even if values are identical (Zustand uses Object.is per-key, but re-reads everything). Calling hide() when already hidden wastes a render cycle. The guard returns early — zero cost, zero renders.

subscribeWithSelector 下任何 set() 都会触发订阅者，即使值相同（Zustand 按键用 Object.is，但仍会重新读取）。已经隐藏时再调 hide() 会浪费一次渲染周期。guard 提前 return → 零代价零渲染。

Quick Decision Cards 决策速查

每次下次遇到类似场景，30 秒内决策。

1. Reuse a component across contexts?

Same UI + same behavior differ only by data/callbacks → reuse + pass currentX=null for "no current" Same UI + different behavior → extract to shared primitive + context-specific wrappers Different UI → don't force reuse

2. Right-click menu: single or dual?

Menu only makes sense when hitting an item (e.g., tree row) → single menu + path param Need different menu on empty space (New/Paste) → dual menu + data-attribute DOM probe Need pre-selection on unselected item → call selectMultiple() inside handler (use ref, NOT state)

3. localStorage boolean view flag

Default OFF (advanced feature opt-in) → read '=== true' + removeItem on exit Default ON (landing page) → tri-state read (null=default) + explicit 'false' write ALWAYS → typeof window === 'undefined' SSR guard Multiple exclusive views → write in resetAllViews for atomicity

4. "Ensure this resource exists" check

Never changes after creation → one-shot flag Can be externally deleted (user on other device) → TTL check (60s for user-paced) Needs real-time state → polling or SSE Has explicit "it changed" signal → cache-then-invalidate

5. Side effect tied to view switch

Effect should almost always run → centralize inside the switch action + opt-out option Effect is rare/special → call explicitly at callsite (opt-in) subscribeWithSelector store action called often → add no-op guard to prevent wasted renders

Q1. You're adding a "Recent files" popover to a new landing page. The existing RecentFilesList component takes currentFileId (for highlighting) and callbacks. On the landing page there's no "current file". What do you do?
你要在新着陆页加一个"最近文件"弹窗。现有的 RecentFilesList 组件接受 currentFileId（用于高亮）和一堆回调。着陆页没有"当前文件"概念。怎么办？

Fork the component into two versions
Reuse it and pass currentFileId={null}; let parent own the fetch/pagination
Rewrite everything from scratch to avoid coupling
Reuse + pass null matches the NewTaskView pattern. Parent owns fetch because the landing page has different pagination/lifecycle needs. Forking creates divergence; rewriting wastes the existing component.

Q2. In your right-click handler, you read selectedSet from a Zustand selector, call selectMultiple([path]), then check selectedSet.has(path). Why is this wrong?
右键处理函数里你从 Zustand selector 读 selectedSet，调用 selectMultiple([path]) 后再 selectedSet.has(path)。为什么错？

Zustand selectors are async
You forgot to pass skipChatSync
selectedSet is a closure-captured value from the last render; it won't update within the same handler call. Use selectedSetRef.current.
You need useSyncExternalStore
Classic React closure trap. The handler captured selectedSet when it was created; calling selectMultiple updates store but the in-scope variable is frozen. Ref captures current via ref.current = state on every render, bypassing the closure.

Q3. You're adding a boolean view flag called isWelcomeView that should default to ON for first-time users, OFF thereafter. What's the right persistence pattern?
你要加一个布尔视图 flag isWelcomeView，首次用户默认开，之后默认关。持久化模式怎么选？

Tri-state read: saved === null ? true : saved === 'true', and explicitly write 'false' on dismiss
Read === 'true' and use removeItem to clean up
Store it in session storage instead
Just use state; don't persist
Option 1 matches NewTaskView. If you used removeItem cleanup, dismissing would delete the key, and the next refresh would read null → default ON again — infinite loop of welcome screen.

Q4. Your app has a "pinned workspace" feature. User A creates a pinned workspace on web, then deletes it on mobile. User A refreshes web — it's still pinned (stale). Which fix?
"Pinned workspace" 功能：用户 A 在 web 创建后在手机上删除。A 刷新 web 仍显示 pinned。怎么修？

Polling every 5s
TTL-based re-verify (e.g., 60s) — cheap, tolerates user-paced external changes
Remove the feature
Only use SSE to sync state
TTL is the right granularity for user-paced external changes. 5s polling would hammer backend with no user benefit (changes are rare). SSE is overkill for "stale on refresh" scenarios. Remove is not a fix.

Q5. You have 5 places calling enterXXXView(). Each one currently calls analytics.track('view_change') after. A new dev adds a 6th view and forgets to track. Best fix?
5 个地方调用 enterXXXView()，每处都手动 analytics.track('view_change')。新人加第 6 个视图时忘了加 track。怎么修？

Add lint rule to warn
Document it in CLAUDE.md
Move the track() call INTO resetAllViews (or the central switching action) with an opt-out option if needed
Nothing; require code reviews
Default-safe centralization. Moving the effect into the callee means forgetting = correct behavior (tracks by default). Lint/docs/reviews rely on humans remembering; centralization doesn't.

Q6. In a Zustand store with subscribeWithSelector, calling set({ isVisible: false }) when isVisible was already false...
Zustand + subscribeWithSelector，已经 isVisible=false 时再调 set({ isVisible: false })...

Does nothing; Zustand short-circuits identical values
Throws an error
Crashes in dev mode
Still triggers the state update internally and may re-notify subscribers; add a no-op guard for hot paths
Zustand doesn't globally short-circuit — setState runs. For hot paths (called many times per second), add if (!get().isVisible) return before set() to avoid wasteful subscriber notifications. This is what W16's hide() guard does.

Flashcards 闪卡 — Click to flip

点击卡片翻面。

Waterfall pagination
A pagination mode where you request { limit, offset, mode: 'waterfall' } and the server returns { data, has_more }. Client keeps a running offset counter and appends results.
瀑布流分页：客户端维护 offset 累加值，server 返回 has_more 决定是否继续。
Closure staleness in event handlers
A handler created in render N captures state from render N. If the state changes in render N+1, the handler still sees the old value. Fix: ref.current = state pattern — ref is mutable and always current.
闭包过期：render N 创建的 handler 捕获 N 时的 state，render N+1 中 state 变了 handler 仍看到旧值。修复：用 ref 同步当前值。
DOM-driven dispatch
Deciding handler behavior from e.target.closest('[data-X]') instead of React state. Useful when the event target itself carries the semantic info (right-click on item vs empty space).
DOM 驱动分派：从 closest('[data-X]') 判断分支，而不是读 React state。当事件目标本身携带语义时很有用。
SSR guard for localStorage
typeof window === 'undefined' check before accessing window.localStorage. Required for Next.js because store initialization runs on both server and client — server has no window.
访问 localStorage 前必加 typeof window === 'undefined' 判定，否则 Next.js SSR 崩溃。
Tri-state localStorage read
For defaults-on-first-use flags: saved === null ? DEFAULT : saved === 'true'. Three states: first-time user (null), explicitly on ('true'), explicitly off ('false'). Prevents "exit → delete → refresh resurrects default".
针对首次默认开启的 flag，三态读取，避免退出删除键后刷新又回到默认开启。
TTL re-check vs one-shot flag
One-shot: boolean that flips true forever after success. TTL: timestamp; re-verify when Date.now() - ts > TTL. Use TTL when external mutation is possible (e.g., user on other device).
一次性 flag 假设成功后状态不变；TTL 允许外部变化恢复，最迟 TTL 秒发现。
Opt-out vs Opt-in side effect
Opt-in: must say "yes" each time → forgetting = unsafe default. Opt-out: default fires, say "no" to skip → forgetting = safe default. Pick based on which direction is "usually correct".
Opt-in 默认不做，容易漏触发；Opt-out 默认触发，漏写时是安全的。根据默认行为哪个更常正确选择。
No-op guard in Zustand action
if (!get().isVisible) return; before set(). Prevents wasted subscriber notifications when the action is called but state wouldn't change. Especially for actions called in loops or on every render.
action 里先判断状态是否需要变，避免 set() 触发无谓订阅者通知。热路径必备。
querySelectorAll for multi-menu outside-click
When you have N popups open simultaneously, register ONE window click listener; inside it, document.querySelectorAll('[data-context-menu]') and iterate. Close only if target is inside none.
多个弹窗共存时，一个 window click 监听器 + querySelectorAll 遍历判断，比每个弹窗各自监听更省。
